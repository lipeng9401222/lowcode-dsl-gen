# 脚本使用指南

> 本文档详细说明 `add_workflow.py`（生成）和 `validate_yml.py`（校验）两个脚本的使用方法。

> ⛔ **路径红线（spec v2）**：`add_workflow.py --app-root` 必须指向 `<apptag>/`，应用资产直接挂在 `<apptag>/` 下；脚本会拒绝任何包含 `/metadata/` 段的路径（fail-fast 退出）。**不要再传 `--metadata` 这个旧别名**——下版本将移除。

## 模式与脚本选择

| 模式 | 场景 | 脚本 |
|---|---|---|
| `create` | 新建/新增/另起一个工作流 | `add_workflow.py` |
| `revise-plan` | 计划或流程图确认阶段纠错 | `workflow_preview.py` + 更新计划，不落应用资产 |
| `update-existing` | 修改已有 `.workflow.yml` | `update_workflow.py` |

`add_workflow.py` 每次都会生成新的 `processguid/processversionguid/version`，只能用于新建。修改已有流程禁止用 `--force` 覆盖重建。

## 快速生成脚本（add_workflow.py）

### 基本用法

```bash
# 最小骨架（开始 + 申请 + 1 个审批 + 结束 + 浏览）
python3 scripts/add_workflow.py --app-root <app-root> --name "<流程名>" --material "<表单名>"

# 多审批节点 + 关联表单数据表（完整入参）
python3 scripts/add_workflow.py \
  --app-root <app-root> \
  --name "费用报销审核流程" \
  --approvers "部门审核,主管审核,财务审核" \
  --material "费用报销申请单" \
  --sql-tablename formtable20260112164507245
```

### 完整参数

| 参数 | 必填 | 类型 | 说明 |
|------|------|------|------|
| `--app-root` | ✅ | 路径 | 应用根目录绝对路径（`<...>/META-INF/resources/<...>/<apptag>/`，**不能含 `/metadata/`**；推荐先 `python3 scripts/path_resolver.py --apptag <tag>` 算好再传） |
| `--name` | ✅ | 字符串 | 流程名称（中文） |
| `--approvers` | 否 | 逗号分隔 | 审批节点列表（不含开始/申请/结束/浏览），默认 `"审批"` |
| `--material` | 否 | 字符串 | 流程材料名称，默认 `"申请表单"` |
| `--sql-tablename` | 否 | 字符串 | 数据库表名（写入 `workflowPvMisTableSet.sql_tablename`） |
| `--author` | 否 | UUID | 创建人 GUID，默认自动生成 |
| `--process-mode` | 否 | 枚举 | `normal/custom/free/subprocess`，默认 `normal` |
| `--methods-json` | 否 | JSON 数组 | 外部方法列表，可含 `ruleguid` 关联动作流 |
| `--events-json` | 否 | JSON 数组 | 流程事件列表，可含 `ruleguid` 关联动作流 |
| `--contexts-json` | 否 | JSON 数组 | 相关数据 workflowContext 列表（spec v2: 用 `fromsqltablename + fromfieldname`，不要写 `fromMisTableId/fromFieldId`） |
| `--conditions-json` | 否 | JSON 数组 | 流转条件列表，可按 `transition` / `transitionguid` / `transitionIndex` 定位；也可按 `fromActivity` / `toActivity` 定位，缺少对应变迁时自动创建 |
| `--activity-materials-json` | 否 | JSON 数组 | 节点字段权限（`workflowActivityMaterial` + `workflowActivityFieldAccess`） |
| `--activity-extra-json` | 否 | JSON 数组 | 节点扩展字段（`overtimeremindcontent` / `smscontent` / `locktimewhenmultitransactor` / `mobileappletrules` / `addattachfilesource` / `joinmethodguid`） |
| `--approve-pass-rate-json` | 否 | JSON 数组 | 审批节点通过率（`passrate` / `passratecalculatemode` / `nopasshandlevalue`） |
| `--free-activities` | 否 | 逗号分隔 | 自由流转节点名；`process-mode=free` 且不传时默认所有审批节点 |
| `--custom-activities` | 否 | 逗号分隔 | 增加“流程设计”按钮的节点名；`process-mode=custom` 且不传时默认第一个审批节点 |
| `--subprocess-guid` | 否 | UUID | `process-mode=subprocess` 时必填，指向子流程 `processguid` |
| `--revoke-option` / `--revoke-allow-day` | 否 | 整数 | 发起人撤销策略 |
| `--no-participator-option` / `--default-user-guid` | 否 | 整数/UUID | 预设处理人不存在时的处理策略 |
| `--filename` | 否 | 字符串 | 自定义文件名（不含扩展名） |
| `--force` | 否 | 标志 | 覆盖已存在文件 |

## 修改已有工作流脚本（update_workflow.py）

修改已有 workflow 时必须原地保留：

- `workflowProcess.processguid`
- `workflowProcessVersion.processversionguid`
- `workflowProcessVersion.version`

基本用法（默认推荐直接传目标节点数组，不依赖 IR）：

```bash
python3 scripts/update_workflow.py \
  --workflow-file <app-root>/workflow/<流程名>.workflow.yml \
  --activities-json '[{"name":"申请","type":"apply"},{"name":"科室负责人审批","type":"approve"}]' \
  --dry-run
```

如果是节点改名，建议显式传改名映射以保留原 activity GUID：

```bash
python3 scripts/update_workflow.py \
  --workflow-file <app-root>/workflow/<流程名>.workflow.yml \
  --activities-json '[{"name":"申请","type":"apply"},{"name":"科室负责人审批","type":"approve","oldName":"部门经理审批"}]' \
  --dry-run
```

确认后去掉 `--dry-run` 写回，再执行：

```bash
python3 scripts/validate_yml.py <app-root>/workflow/<流程名>.workflow.yml
```

## 流程图预览脚本（workflow_preview.py）

规划确认和纠偏时必须输出 Mermaid 简易流程图，只展示业务节点，不展示“开始/结束/浏览”：

```bash
python3 scripts/workflow_preview.py --activities-json '[{"name":"申请","type":"apply"},{"name":"科室负责人审批","type":"approve"}]'
python3 scripts/workflow_preview.py --from-ir .lowcode-plans/<apptag>/ir.yml --asset-id <asset-id>
python3 scripts/workflow_preview.py --workflow-file <app-root>/workflow/<流程名>.workflow.yml
```

`--from-ir` 仅用于兼容旧计划或审计流程；单 workflow 创建和节点链路纠偏默认使用 `--activities-json`。

### 工作流能力示例

```bash
# 自定义流程：第一个审批节点自动增加“流程设计”按钮
python3 scripts/add_workflow.py --app-root <app-root> --name "合同审批流程" --process-mode custom

# 自由流程：审批节点禁用退回按钮，并设置自由流转模式
python3 scripts/add_workflow.py --app-root <app-root> --name "领导批示流程" --process-mode free --approvers "领导批示"

# 主子流程：主流程插入 activitytype=90 子流程节点
python3 scripts/add_workflow.py \
  --app-root <app-root> \
  --name "报销主流程" \
  --process-mode subprocess \
  --subprocess-guid <子流程processguid>

# 事件或外部方法关联动作流 ruleguid
python3 scripts/add_workflow.py \
  --app-root <app-root> \
  --name "审批流程" \
  --events-json '[{"eventname":"审批后更新状态","belongto":22,"eventtype":55,"sourceOperation":"审批:送下一步","ruleguid":"updateStatusActionFlow"}]'
```

### 条件分支示例

“否则 / 不满足条件”不要用默认变迁表达，必须生成显式互补条件。下面示例表示：需求来源为内部优化时走“总负责人评审”，否则从“申请”直接到“指定人员执行”。脚本会自动补 `申请 → 指定人员执行` 这条 transition，并把“总负责人评审”放到分支泳道。

```bash
python3 scripts/add_workflow.py \
  --app-root <app-root> \
  --name "需求评审流程" \
  --approvers "总负责人评审,指定人员执行" \
  --sql-tablename demandinfo \
  --contexts-json '[{"fieldname":"demandsource","note":"需求来源","fieldtype":10,"valuesource":30,"fromfieldname":"demandsource"}]' \
  --conditions-json '[{"fromActivity":"申请","toActivity":"总负责人评审","transitionname":"总负责人评审","conditionname":"需求来源为内部优化","leftvalue":"[#=demandsource#]","lefttext":"需求来源","compareoperation":"=","rightvalue":"4","righttext":"内部优化","valuetype":10},{"fromActivity":"申请","toActivity":"指定人员执行","transitionname":"直接执行","conditionname":"需求来源非内部优化","leftvalue":"[#=demandsource#]","lefttext":"需求来源","compareoperation":"!=","rightvalue":"4","righttext":"非内部优化","valuetype":10}]'
```

### 复杂条件示例

`conditionexpressiontype=20` 表示复杂表达式记录，用 `conditionexpression` 组合同一条 transition 下已有的 `conditionexpressiontype=10` 单一表达式。`[1]`、`[2]` 是这些单一表达式按 `ordernum` 升序、再按列表顺序排列后的 1-based 序号，不能跨 transition 引用。

下面示例表示：申请人年龄大于 20 且性别为男时，才从“申请”流转到“人工复核”。

```bash
python3 scripts/add_workflow.py \
  --app-root <app-root> \
  --name "人员复核流程" \
  --approvers "人工复核" \
  --sql-tablename personinfo \
  --contexts-json '[{"fieldname":"age","note":"年龄","fieldtype":30,"valuesource":30,"fromfieldname":"age"},{"fieldname":"gender","note":"性别","fieldtype":10,"valuesource":30,"fromfieldname":"gender"}]' \
  --conditions-json '[{"fromActivity":"申请","toActivity":"人工复核","transitionname":"人工复核","conditionname":"年龄大于20","conditionexpressiontype":10,"leftvalue":"[#=age#]","lefttext":"年龄","compareoperation":">","rightvalue":"20","righttext":"20","valuetype":30,"ordernum":0},{"fromActivity":"申请","toActivity":"人工复核","transitionname":"人工复核","conditionname":"性别是男","conditionexpressiontype":10,"leftvalue":"[#=gender#]","lefttext":"性别","compareoperation":"=","rightvalue":"male","righttext":"男","valuetype":10,"ordernum":1},{"fromActivity":"申请","toActivity":"人工复核","transitionname":"人工复核","conditionname":"年龄大于20且性别男","conditionexpressiontype":20,"conditionexpression":"[1] AND [2]","ordernum":2}]'
```

强校验规则：

- `conditionexpressiontype=10` 必须包含 `leftvalue`、`compareoperation`、`rightvalue`、`valuetype`。
- `conditionexpressiontype=20` 必须包含 `conditionexpression`，且 `[n]` 能解析到同 transition 下的 type=10 条件。
- `conditionexpressiontype=30` 必须包含 `methodguid` 或 `ruleguid`。

### 对话脚本（创建简单顺序流）

#### 第 1 步：流程基本信息

```text
你要创建什么工作流？我需要知道：
1. 流程名称（中文，如 采购立项审核流程）
2. 是否 Vue 流程？建议 0（非 Vue）
3. 流程级时限要不要限制？默认 -1 不限
```

#### 第 2 步：审批节点（不含开始/申请/结束/浏览）

```text
请描述审批节点（按顺序，逗号分隔，不需要写"开始""申请""结束""浏览"）：
节点 1 名称（如 部门审核）：
节点 2 名称（如 主管审批）：
节点 3 名称（如 财务审核）：

简单顺序流：每个相邻节点直接连接，无条件
条件分支（可选）：金额 <= 500 走"主管审批"；金额 > 500 走"总监审批"。若出现"否则/不满足条件"，需准备互补条件；若需要跨过中间节点直达后续节点，在条件项中写 `fromActivity` / `toActivity`
```

#### 第 3 步：表单材料

```text
该流程关联的表单是？我需要：
1. 表单名称（如 采购立项申请表）
2. 数据库表名（用于 workflowPvMisTableSet.sql_tablename，必须能匹配同应用 mis.tableName）
3. 表单页 / 浏览页 pagetag（已有 pagedesigne 时脚本可自动推断，多个页面时需明确选择）
```

#### 第 4 步：更新计划与追溯 + 明确批准

- 所有 GUID 自动生成 UUIDv4
- 节点坐标自动按动态横向布局生成：开始 `(0,204)`、申请 `(235,180)`、后续节点按卡片估算宽度递增且最小间距不低于 `320px`、浏览 `(24,-34)`
- 流程材料 PC 地址自动关联 pagedesigne：`pageurl_read=detail?pagetag=<详情页>`、`pageurl_readandwrite=add?pagetag=<表单页>`，移动端材料地址暂不填充
- 自动生成开始 + 申请 + 审批节点 + 结束 + 浏览 5 类节点
- 每个审批节点自动生成"通过 + 退回"按钮
- 每个退回按钮自动生成对应 workflowConfig
- 先展示生成清单和目标文件路径
- 计划文档必须记录每轮流程能力、节点、表单材料、外部方法和动作流引用问题、选项、用户回复和确认结果
- 用户明确回复"批准创建"后，再调用脚本一键生成并校验

## 校验脚本（validate_yml.py）

### 基本用法

```bash
# 校验单个工作流文件
python3 scripts/validate_yml.py <app-root>/workflow/<流程名>.workflow.yml

# 校验整个应用（含跨引用关系）
python3 scripts/validate_yml.py --check-refs <app-root>

# spec v2 严格模式（强烈推荐用于交付前最终校验）
# 已废弃字段 / metadata 残留 / 旧驼峰键 / tableid 残留 一律升级为 error
python3 scripts/validate_yml.py --strict --check-refs <app-root>
```

### 通过条件

**错误数 = 0** 才算交付。出现任何 error 必须修复后重新校验，警告（warn）也建议处理。

### `--strict` 模式具体会拦下什么

| 类别 | 命中举例 | 默认 | --strict |
|------|---------|------|---------|
| 旧驼峰键 | `isDefault / mobileHandleType / isLockWhenMultiTransactor / subProMultiTransctorMode / isShowLineGraph / revokeOption ...` | warn | **error** |
| 已废弃字段 | `tableid` / `fromMisTableId` / `fromFieldId` 残留 | warn | **error** |
| 老结构残留 | `<apptag>/metadata/` 目录或路径段 | warn | **error** |
| 坐标 / 默认值偏离 | 坐标重叠、浏览节点非 `(24,-34)`、`splittype/is_checkmaterialsubmit/operationvisiablecase` 不合默认 | warn | 非 0 |
| 条件分支不完整 | 同一源节点部分 outgoing 有条件、部分无条件；条件 transition 仍设 `is_default=10`；可选分支节点未放到 `icony='20'` | warn | 非 0 |
| 材料页面地址异常 | `pageurl_read` 未走 detail 或 `pageurl_readandwrite` 未走 add | warn | 非 0 |
| 引用未闭合 | transition.from/to 找不到、退回按钮缺 workflowConfig 等 | error | error |
| workflowContext valuesource=30 必填项缺失 | 缺 `frommaterialguid / fromsqltablename / fromfieldname` | error | error |

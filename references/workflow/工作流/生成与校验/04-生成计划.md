# 04 生成计划

目标：在落盘前让用户明确将要生成或修改什么。

> 计划门禁的唯一硬约束（批准前只写 `.lowcode-plans/<apptag>-plan.md`、不调脚本、不写 metadata；AI 不判断宿主是否暴露 `request_user_input`/`/plan`）见 `SKILL.md` 的「计划文档门禁」节。本文档仅补充「生成计划」阶段的阶段状态跟踪与计划模板。

## 交互原则

- 批准确认不用 `y/n`；以语义表达为主（批准生成 / 调整计划 / 暂不生成）。
- 每轮最多 3 个问题，且只问当前阶段；不要把 codeitem、mis、module、pagedesigne、event、workflow 的问题一次性混在一起。

## 分阶段确认状态

计划文档必须显式记录阶段状态，避免模型跳步：

- `current_stage`：当前正在确认的阶段。
- `confirmed_stages`：已经由用户确认的阶段。
- `pending_stages`：尚未确认的阶段，按主体架构 → 基本信息 → codeitem → mis → module → pagedesigne → workflow → event → 最终批准排序。
- `next_question`：下一步只问什么。
- `stage_note`：当前阶段的简短摘要，不把未确认信息写成确定事实。

## 计划格式

```markdown
---
tool_name: "<Codex CLI | Claude Code | Windsurf | Cursor | CodeBuddy | Kiro | Trae | Cline/Roo Code | Aider | Gemini CLI | Qwen Code | unknown-cli-agent>"
current_stage: "<主体架构 | 基本信息 | codeitem | mis | module | pagedesigne | workflow | event | 最终批准>"
confirmed_stages: []
pending_stages: ["主体架构", "基本信息", "codeitem", "mis", "module", "pagedesigne", "workflow", "event", "最终批准"]
next_question: "<下一步只问当前阶段的 1~3 个问题>"
approval_status: pending
approval_text: ""
---

我准备按下面的计划生成低代码应用：

- 应用：<applicationname> (<apptag>)
- 计划文档：.lowcode-plans/<apptag>-plan.md
- 当前阶段：<current_stage>
- 已确认阶段：<confirmed_stages>
- 待确认阶段：<pending_stages>
- 下一步只问：<next_question>
- 目录：<app-root>
- 默认值：
  - developerstag: epoint
  - baseouguid: 空（仅用户明确要求时填 epoint 或其他值）
- 新增资产：
  - codeitem: <名称列表>
  - mis: <表名列表>
  - module: <模块列表>
  - event: <动作流列表；未要求时写“仅保留 event/ 目录，不生成 .event.yml”>
  - workflow: <流程列表>
  - pagedesigne: <页面列表>
- 待确认：
  - codeitem: <缺少子项时列出问题>
  - mis: <缺少表名/字段清单时列出问题>
- 会运行的脚本：
  - scripts/scaffold_app.py ...
  - scripts/add_pagedesigne.py ...
- 校验：
  - scripts/validate_yml.py --strict --check-refs <app-root>

请回复"批准创建"或说明要调整哪一项。
```

## 约束

- 没有明确批准前不写应用资产文件；批准前唯一允许写入的文件是 `.lowcode-plans/<apptag>-plan.md`。
- **路径红线（spec v2）**：计划文档里"目标目录 / 应用根 / app-root"必须等于 `python scripts/path_resolver.py --apptag <tag>` 的输出，且**不允许**包含 `/metadata/` 段。出现 `<apptag>/metadata/<asset>/...` 这种老结构路径时，计划无法被批准——脚本会 fail-fast 拦下。**禁止**在计划里使用"metadata 目录"等措辞，统一写"应用根目录"。
- 如果目标文件已存在，必须在计划里标注"修改/覆盖/跳过"的策略；对 `--force` 的使用必须明确说明。
- `approval_status` 只有在用户明确回复"批准创建 / 按计划生成 / 可以落盘"等批准语义后才能改为 `approved`，并把原文记录到 `approval_text`。
- `codeitem` 未确认子项或 `mis` 未确认字段清单时，不得把对应脚本列入"会运行的脚本"。
- 未明确要求动作流时，不得把 `scripts/add_event.py` 列入"会运行的脚本"。
- 未明确要求工作流时，不得把 `scripts/add_workflow.py` 列入"会运行的脚本"。

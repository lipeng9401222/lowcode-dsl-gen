# 04 生成计划与对话追溯

目标：在落盘前让用户明确将要生成或修改什么，并把确认过程完整保存在 `.lowcode-plans/<apptag>-plan.md` 中，方便后续追溯。

> 计划与追溯门禁的唯一硬约束（第一轮就写 `.lowcode-plans/<apptag>-plan.md`、每轮记录问题/选项/回复/确认结果、批准前不调脚本不写应用资产；AI 不判断宿主是否暴露 `request_user_input`/`/plan`）见 `SKILL.md` 的「计划文档与对话追溯门禁」节。本文档补充「生成计划」阶段的阶段状态跟踪、对话记录格式与计划模板。

## 交互原则

- 批准确认不用 `y/n`；以语义表达为主（批准生成 / 调整计划 / 暂不生成）。
- 每轮最多 3 个问题，且只问当前阶段；不要把 codeitem、mis、module、pagedesigne、event、workflow 的问题一次性混在一起。
- 提问前先写计划文档的 `对话确认记录`，用户回复后回填同一条记录；Plan Mode / 结构化问答也必须同步记录。
- 用户主要审阅可读计划草案；`ir.yml` 是机器缓存和审计材料。默认不向用户展开完整 IR，除非用户要求查看 IR、调试失败或做审计追溯。

## 分阶段确认状态与追溯字段

计划文档必须显式记录阶段状态，避免模型跳步：

- `plan_revision`：每次更新计划文档递增。
- `created_at` / `updated_at`：计划创建与最后更新时间。
- `last_interaction_id`：最近一次对话确认记录编号。
- `current_stage`：当前正在确认的阶段。
- `confirmed_stages`：已经由用户确认的阶段。
- `pending_stages`：尚未确认的阶段，按主体架构 → 基本信息 → codeitem → mis → module → pagedesigne → workflow → event → 最终批准排序。
- `next_question`：下一步只问什么。
- `stage_note`：当前阶段的简短摘要，不把未确认信息写成确定事实。
- `approval_status` / `approval_text`：最终批准状态和用户批准原文。
- `ir_path`：固定为 `.lowcode-plans/<apptag>/ir.yml`，禁止 `.lowcode-plans/<apptag>-ir.yml`。
- `ir_visibility`：默认 `hidden`，仅在用户要求、调试或审计时改为 `visible`。

每条 `对话确认记录` 至少包含：

- `interaction_id`
- `stage`
- `ask_mode`：`natural_language` / `plan_mode` / `structured_question`
- `question`
- `options`
- `recommended_option`
- `status`：提问前为 `awaiting_user`，回复后改为 `confirmed` / `adjusted` / `ambiguous` / `rejected` / `pending`
- `user_reply`
- `parsed_result`
- `stage_effect`

  - tenantguid: <user_confirmed + questionId | repo_inferred + repoEvidence | safe_default + reason>

## 资产子计划

每个资产必须有子计划，路径为 `.lowcode-plans/<apptag>/<asset-type>/<asset-id>-plan.md`，且至少包含：

- `status: <pending|generating|validating|done|failed|skipped>`
- 输入 IR 摘要
- 确认项与来源
- dry-run 命令和结果
- 生成后校验命令和结果

`validate_plan.py` 会审计上述内容。只有 `status` 和一段“待 IR open_questions 解决后生成”的占位子计划必须失败。

workflow 子计划额外必须包含实现效果确认矩阵：

| 确认项 | 状态 | 来源 | 说明 |
|---|---|---|---|
| 流程目标 | confirmed | user_confirmed/repo_inferred | 业务效果 |
| 节点链路 | confirmed | user_confirmed/repo_inferred | 申请、审批、分支 |
| 处理人来源 | confirmed | user_confirmed/repo_inferred | 指定人/角色/组织 |
| 条件分支 | confirmed/not_required | user_confirmed/repo_inferred | 条件与互补条件 |
| 表单/详情页 pagetag | confirmed | user_confirmed/repo_inferred | pageTagForm/pageTagDetail |
| 关联 MIS 表 | confirmed | user_confirmed/repo_inferred | sqlTableName |
| 字段权限 | confirmed/not_required | user_confirmed/repo_inferred | activityMaterials |
| 会签/通过率 | confirmed/not_required | user_confirmed/repo_inferred | approvePassRate |
| 超时提醒 | confirmed/not_required | user_confirmed/repo_inferred | activityExtra |
| 状态联动/event | confirmed/not_required | user_confirmed/repo_inferred | methods/events/ruleguid |

workflow 子计划还必须包含活动转换表，以及 conditions/contexts/methods/events 摘要。

workflow 子计划还必须包含：

- `operationMode` 判定依据：`create` / `revise-plan` / `update-existing`，以及触发证据。
- 流程图预览：Mermaid `flowchart`，只展示业务节点和条件分支，不展示“开始/结束/浏览”。
- 用户纠偏记录：记录用户指出的问题、修改前后节点/分支；无纠偏时写“无”。
- 修改已有流程时的 GUID 保留声明：`processguid`、`processversionguid`、`version` 均保持不变。

## 约束

- 没有明确批准前不写应用资产文件；批准前唯一允许写入的文件是 `.lowcode-plans/<apptag>-plan.md`（或 apptag 未定时的临时任务简称计划）。
- 如果工程根尚未定位，第一轮只问定位问题；工程根确认后，必须把这轮定位问题、选项和用户回复补录进计划文档。
- 每轮提问前必须先追加一条 `对话确认记录`，状态为 `awaiting_user`；收到用户回复后必须回填同一条记录，再推进阶段。
- 如果宿主提供 Plan Mode / 结构化问答，问题、选项、用户选择和自由文本仍必须写入计划文档。
- 确认 apptag 后，必须把临时计划迁移为 `.lowcode-plans/<apptag>-plan.md`，并保留所有早期对话确认记录。
- **路径红线（spec v2）**：计划文档里"目标目录 / 应用根 / app-root"必须等于 `python3 scripts/path_resolver.py --apptag <tag>` 的输出，且**不允许**包含 `/metadata/` 段。出现 `<apptag>/metadata/<asset>/...` 这种老结构路径时，计划无法被批准——脚本会 fail-fast 拦下。**禁止**在计划里使用"metadata 目录"等措辞，统一写"应用根目录"。
- 如果目标文件已存在，必须在计划里标注"修改/覆盖/跳过"的策略；对 `--force` 的使用必须明确说明。
- 同名 workflow 已存在且用户语义模糊时，必须先确认"新增独立流程"还是"修改原流程"，推荐"修改原流程"。
- `revise-plan` 或 `update-existing` 禁止列出 `add_workflow.py --force`；修改已有 workflow 必须列出 `scripts/update_workflow.py`。
- 修改已有 workflow 时，计划必须写明不改变 `processguid/processversionguid/version`。
- `approval_status` 只有在用户明确回复"批准创建 / 按计划生成 / 可以落盘"等批准语义后才能改为 `approved`，并把原文记录到 `approval_text`。
- `approval_status=approved` 时，主计划不得再出现“待确认/需确认/待解决/pending open questions”等语义。
- `codeitem` 未确认子项或 `mis` 未确认字段清单时，不得把对应脚本列入"会运行的脚本"。
- 未明确要求动作流时，不得把 `scripts/add_event.py` 列入"会运行的脚本"。
- 未明确要求工作流时，不得把 `scripts/add_workflow.py` 列入"会运行的脚本"。

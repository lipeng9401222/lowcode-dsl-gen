# 主子流程

## 适用场景

把**可复用、可独立管控**的业务环节从主流程中拆出，封装成独立的子流程，再由主流程通过子流程节点调用。常见诉求：

- 总体流程相似、仅因部门/层级不同导致部分步骤不同（用子流程做"动态绑定"）
- 同一业务环节（如发票核验、专家评审、电子签章、权限配置）在多个主流程中复用
- 主流程节点过多、想做"分层设计"降低复杂度

## 业务概念界定

| 维度 | 主流程 YAML | 子流程 YAML |
|------|------------|------------|
| 定义 | 完整业务全生命周期的流程定义（开始 → 申请 → … → 结束） | 主流程中某一特定环节的精细化拆分与封装 |
| 角色 | 业务流程的"主干"，可独立运行、闭环管理 | 主流程的"分支模块"，专注解决单一可复用环节 |
| 独立性 | 可直接关联业务主体（工单/合同/报销单等），可独立发起 | 不能独立发起，仅由主流程的子流程节点触发 |
| 完整性 | 必须含开始(10) + 申请(30) + N 审批(30) + 结束(20) + 浏览(100) | 同样必须是一份完整的 `workflow.yml`（不能写成片段） |

## 典型业务场景

| 主流程 | 拆出的子流程 | 子流程作用 |
|--------|--------------|-----------|
| 费用报销全流程 | 发票核验子流程 | 封装"发票上传 → 真伪校验 → 金额匹配 → 通过/驳回"，可复用至差旅、采购等各类报销 |
| 项目立项全流程 | 专家评审子流程 | 封装"分配专家 → 下发材料 → 线上打分 → 汇总结论"，多类立项主流程复用 |
| 合同签订全流程 | 电子签章子流程 | 封装"签章权限校验 → PDF 生成 → 签章触发 → 结果校验 → 日志"，跨合同类型复用 |
| 员工入职全流程 | 权限配置子流程 | 封装"岗位权限匹配 → 账号创建 → 赋权 → 验证 → 生效"，新员工/调岗复用 |

## 关键配置

主流程通过 `activitytype=90` 节点调用子流程；该节点通过 `callsubprocessguid` 绑定子流程的 `processguid`。

```yaml
# 主流程（费用报销主流程）
type: workflow
workFlow:
  workflowProcess:
    processguid: <主流程 processguid>
    processname: "费用报销主流程"
    designversion: senior
    status: 10
    tag: '20'

  workflowVersion:
    activity:
      # ... 开始 / 申请 / 审批 节点（略，结构同 01-简单审批流）...

      # 子流程节点（关键）
      - activityguid: <UUID-子流程节点>
        activityname: "发票核验子流程"
        activitydispname: "发票核验子流程"
        processversionguid: <主流程版本 UUID>
        activitytype: 90                       # 子流程节点类型
        splittype: 20
        jointype: 30
        callsubprocessguid: <子流程 processguid>  # 必须是已存在子流程的 processguid
        subprocesssynctype: 20                 # 10 异步；20 同步（等待子流程完成）
        multitransactormode: 10                # 10 全部子流程完成；20 任一完成
        subProMultiTransctorMode: 10           # 10 多处理人传给子流程第一个活动；20 各自独立处理
        iconx: '545.5'
        icony: '99'
        vmlid: 4

      # ... 结束 / 浏览 节点（略）...
```

## 联动核心规则

1. **触发规则**：主流程通过 `activitytype=90` 子流程节点触发子流程；触发条件由主流程上游变迁决定（如"财务审核通过后触发发票核验"）。
2. **数据交互**：子流程从主流程获取业务上下文（报销单 ID、发票编号等），执行完成后将结果（核验通过/驳回、评审分数等）回调到主流程指定字段。
3. **同步 vs 异步**：
   - `subprocesssynctype=20`（同步）：主流程等待子流程完成后再继续流转
   - `subprocesssynctype=10`（异步）：子流程并行执行，主流程通过结果回调接收状态
4. **异常兜底**：子流程失败/超时/驳回时，主流程需通过变迁条件兜底（如"核验驳回则回退到员工提报节点"）。
5. **复用规则**：同一子流程可被多个主流程调用，需保证子流程入参/出参通用，避免与特定主流程强耦合。
6. **状态关联**：主流程同步子流程执行状态（"运行中-发票核验中"、"已完成-发票核验通过"等），便于整体监控。

## 选型建议（主流程 vs 子流程）

| 业务特征 | 优先放主流程 | 优先拆子流程 |
|----------|--------------|--------------|
| 流程范围 | 覆盖业务全生命周期 | 仅单一业务环节 |
| 复用性 | 仅特定业务专属 | 多业务场景复用 |
| 复杂度 | 节点少、逻辑简单 | 节点多、规则复杂（需封装） |
| 独立性 | 可独立发起、闭环运行 | 依赖主流程上下文，无法独立运行 |
| 数据交互 | 仅内部数据流转 | 需与主流程双向数据交互 |

## 生成要点

- `callsubprocessguid` 必须是**已存在子流程**的 `workflowProcess.processguid`，引擎会在发布时校验。
- 子流程**必须是完整的 `workflow.yml`**（含开始/申请/审批/结束/浏览 5 类节点），不能只写片段。
- 主、子两份流程文件落在同一应用根目录下，建议命名：`<主流程名>.workflow.yml` + `<子流程名>.workflow.yml`。
- 同步/异步的选择：业务强一致选同步（`20`），可并行可后置选异步（`10`）。
- 子流程节点本身**没有 `workflowActivityOperation`**（按钮）；流转通过子流程内部完成。

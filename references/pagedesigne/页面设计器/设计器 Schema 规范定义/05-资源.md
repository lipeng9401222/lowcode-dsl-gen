# 资源

`resources` 用于声明页面依赖的外部能力，例如接口、选项、树数据、上传入口、路由目标和工作流服务。它只描述“有什么能力”，不描述“何时调用”；调用时机由 `actions`、节点 `source` 或运行时导航能力决定。

## 资源定位

`resources` 是顶层注册区，负责集中声明页面依赖的外部能力：

- 节点通过 `source` 直接引用资源 id。
- action step 通过 `use: "resourceId.operationId"` 调用具备命名操作的资源。
- 工作流 step 通过 `workflow.use: "resourceId.operationId"` 调用工作流资源能力。
- 导航 step 通过 `navigate` 或 `navigate.to` 引用 `route` 资源。
- 资源本身不保存接口响应、缓存、上传进度或其他运行时状态。

这意味着 `resources` 更像“能力目录”，而不是运行时请求日志。Schema 只需要表达“某个节点的选项来自哪里”“某个动作调用哪个后端操作”“某个按钮跳转到哪个目标”，不在这里规定加载时机、重试策略或错误提示方式。

下面是一个总览示例，展示不同资源在同一页中的典型组合方式：

```json
{
  "resources": {
    "deptOptions": {
      "type": "dictionary",
      "source": "remote",
      "request": { "method": "GET", "url": "/api/departments" },
      "map": { "label": "name", "value": "id" }
    },
    "employeeApi": {
      "type": "endpoint",
      "baseUrl": "/api/employees",
      "operations": {
        "list": { "method": "GET", "path": "" },
        "save": { "method": "POST", "path": "" }
      }
    },
    "avatarUpload": {
      "type": "upload",
      "request": { "method": "POST", "url": "/api/files" },
      "map": { "url": "url", "id": "id", "name": "filename" }
    }
  }
}
```

## 通用结构

所有资源都属于 `ResourceDefinition` 联合类型，`type` 用于区分具体资源类别：

```ts
type ResourceDefinition =
  | EndpointResource
  | DictionaryResource
  | TreeResource
  | UploadResource
  | RouteResource
  | WorkflowResource;
```

涉及 HTTP 请求的资源会复用 `RequestDefinition`：

```ts
interface RequestDefinition {
  method: 'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE';
  url?: string;
  path?: string;
  params?: Record<string, ExpressionString>;
  body?: ExpressionString;
}
```

- `method`：请求方法，必填。
- `url`：完整请求地址，适合独立请求资源，例如远程字典、树加载、文件上传。
- `path`：相对路径，适合挂在 `endpoint.baseUrl` 或其他具备基础地址语义的资源下。
- `params`：请求参数映射，值是表达式字符串，可从 `model.*`、`params.*`、`response.*` 等上下文取值。
- `body`：请求体表达式，通常用于提交对象、表单或工作流参数。

`url` 和 `path` 并不是二选一的强校验关系；Schema 层只负责描述请求意图，具体拼接方式由消费运行时决定。推荐做法是：独立请求优先使用 `url`，一组同域 API 优先通过 `baseUrl + path` 组织。

## 资源类型

### endpoint

`endpoint` 用于声明一组可命名的通用后端操作，适合列表查询、详情加载、保存、删除这类复用型业务接口。它本身不表示“立即发起请求”，而是给 `actions.steps[].use` 提供稳定的调用入口。

字段说明：

- `baseUrl`：这组操作共享的基础地址。
- `operations`：命名操作集合，key 是操作名，value 是 `RequestDefinition`。

最小示例：

```json
{
  "employeeApi": {
    "type": "endpoint",
    "baseUrl": "/api/employees",
    "operations": {
      "list": { "method": "GET", "path": "" },
      "detail": { "method": "GET", "path": "/detail" },
      "save": { "method": "POST", "path": "" }
    }
  }
}
```

联动示例：

```json
{
  "actions": {
    "refreshEmployees": {
      "steps": [
        {
          "use": "employeeApi.list",
          "params": {
            "keyword": "model.search.keyword"
          },
          "assign": {
            "model.employeeList": "response.items"
          }
        }
      ]
    }
  }
}
```

推荐写法：

- 当多个操作共享同一域名前缀时，优先使用一个 `endpoint` 聚合。
- `operations` 中的 key 应表达业务动作语义，例如 `list`、`detail`、`save`，避免使用无意义编号。

### dictionary

`dictionary` 面向平铺选项数据，常用于下拉框、单选、多选等 `label/value` 结构的数据源。它可以是静态选项，也可以是远程请求结果。

字段说明：

- `source`：`static` 或 `remote`。
- `options`：静态选项数组，通常在 `source = "static"` 时使用。
- `request`：远程选项请求定义，通常在 `source = "remote"` 时使用。
- `map`：把响应中的字段映射成统一的 `label/value` 语义。

静态选项示例：

```json
{
  "genderOptions": {
    "type": "dictionary",
    "source": "static",
    "options": [
      { "label": "男", "value": "M" },
      { "label": "女", "value": "F" }
    ]
  }
}
```

远程选项示例：

```json
{
  "deptOptions": {
    "type": "dictionary",
    "source": "remote",
    "request": {
      "method": "GET",
      "url": "/api/departments"
    },
    "map": {
      "label": "name",
      "value": "id"
    }
  }
}
```

联动示例：

```json
{
  "type": "select",
  "model": "employee.deptId",
  "textModel": "employee.deptName",
  "source": "deptOptions",
  "props": { "label": "部门" }
}
```

推荐写法：

- 少量固定选项优先用 `static`，减少不必要的请求依赖。
- 远程字典优先通过 `map` 显式声明 `label/value` 来源，不要假设运行时总能猜中字段名。

### tree

`tree` 用于声明树形选择数据源，适合部门树、行政区划树、菜单树等层级结构。它和 `dictionary` 一样可以被节点通过 `source` 引用，但返回语义是层级数据而不是平铺选项。

字段说明：

- `request`：树数据请求定义。
- `map.label`：节点文本字段名。
- `map.value`：节点值字段名。
- `map.children`：子节点数组字段名。

最小示例：

```json
{
  "deptTree": {
    "type": "tree",
    "request": {
      "method": "GET",
      "url": "/api/departments/tree"
    },
    "map": {
      "label": "name",
      "value": "id",
      "children": "children"
    }
  }
}
```

联动示例：

```json
{
  "type": "treeSelect",
  "model": "employee.deptId",
  "textModel": "employee.deptName",
  "source": "deptTree",
  "props": { "label": "所属部门" }
}
```

推荐写法：

- 当后端字段名不固定时，总是补齐 `map`，避免运行时依赖隐式约定。
- 只要数据本质上是层级结构，就优先用 `tree`，不要把树形接口包装成普通 `dictionary`。

### upload

`upload` 用于声明文件或图片上传入口。它负责描述上传请求和上传结果映射，不负责维护文件列表、上传进度或预览状态。

字段说明：

- `request`：上传请求定义，必填。
- `accept`：允许的文件类型列表，例如 `["image/png", "image/jpeg"]`。
- `maxSizeMb`：单文件大小上限，单位 MB。
- `map`：把上传响应中的字段映射为运行时常用文件信息。

最小示例：

```json
{
  "avatarUpload": {
    "type": "upload",
    "request": {
      "method": "POST",
      "url": "/api/files/avatar"
    },
    "accept": ["image/png", "image/jpeg"],
    "maxSizeMb": 5,
    "map": {
      "url": "url",
      "id": "id",
      "name": "filename"
    }
  }
}
```

联动示例：

```json
{
  "type": "upload",
  "model": "employee.avatarId",
  "source": "avatarUpload",
  "props": { "label": "头像" }
}
```

推荐写法：

- `accept` 和 `maxSizeMb` 只表达协议级限制，真正的拦截和提示由运行时实现。
- 当上传响应字段名与运行时默认约定不一致时，补齐 `map`，避免文件回填不完整。

### route

`route` 用于声明页面、弹窗或外部链接等导航目标。它表达的是“去哪里”，不是“调用哪个 HTTP 接口”。

字段说明：

- `path`：目标路径，必填。
- `params`：动态参数映射，值为表达式字符串。

最小示例：

```json
{
  "employeeDetailRoute": {
    "type": "route",
    "path": "/employees/detail",
    "params": {
      "id": "model.employee.id"
    }
  }
}
```

联动示例：

```json
{
  "actions": {
    "openEmployeeDetail": {
      "steps": [
        { "navigate": "employeeDetailRoute" }
      ]
    }
  }
}
```

推荐写法：

- 把稳定复用的页面目标抽成 `route` 资源，避免在多个动作中重复硬编码路径。
- 当参数依赖当前模型、列表行或页面入参时，用 `params` 显式声明来源。

### workflow

`workflow` 用于声明工作流服务入口，适合发起流程、查询流程状态、执行审批动作等场景。它和通用 `endpoint` 的区别不在于请求结构，而在于资源语义面向流程能力。

字段说明：

- `processKey`：流程定义标识。
- `operations`：可选的命名操作集合，便于 action 的 `workflow.use` 统一引用。

最小示例：

```json
{
  "employeeWorkflow": {
    "type": "workflow",
    "processKey": "employee_onboarding"
  }
}
```

扩展示例：

```json
{
  "employeeWorkflow": {
    "type": "workflow",
    "processKey": "employee_onboarding",
    "operations": {
      "start": {
        "method": "POST",
        "path": "/start",
        "body": "model.employee"
      },
      "approve": {
        "method": "POST",
        "path": "/approve",
        "body": "params.approvalPayload"
      }
    }
  }
}
```

联动示例：

```json
{
  "actions": {
    "startEmployeeWorkflow": {
      "steps": [
        {
          "workflow": {
            "use": "employeeWorkflow.start",
            "businessKey": "model.employee.id",
            "payload": "model.employee"
          }
        }
      ]
    }
  }
}
```

推荐写法：

- 如果页面只需要标识某个流程定义，保留 `processKey` 即可。
- 如果一个页面会多次触发不同工作流动作，再补 `operations`，让引用点保持稳定短路径。

## 映射与约定

多个资源类型会通过 `map` 把后端响应字段映射到运行时约定语义：

- `ResourceMap.label`：选项或节点文本字段。
- `ResourceMap.value`：选项或节点值字段。
- `TreeResourceMap.children`：树节点子项数组字段。
- `UploadResourceMap.url`：上传后可访问地址字段。
- `UploadResourceMap.id`：文件唯一标识字段。
- `UploadResourceMap.name`：文件显示名称字段。

`map` 不是脚本转换器，也不负责做复杂对象重组。它只声明“运行时应该从响应对象的哪个字段取值”。如果后端字段名已经与运行时约定一致，某些 `map` 可以省略；如果字段名不一致，推荐显式补齐。

## 最佳实践与校验提示

- 节点的 `source` 必须引用存在且类型兼容的资源 id。
- `actions.steps[].use` 必须引用存在的 `resource.operation`。
- `actions.steps[].workflow.use` 也应引用存在的工作流资源操作。
- `endpoint` 适合组织一组共享基础地址的命名操作；单个独立请求通常直接写在 `dictionary`、`tree`、`upload` 或 `workflow` 的 `request` / `operations` 中更清晰。
- `dictionary` 适合平铺选项，`tree` 适合层级数据，不要混用。
- `route` 负责导航目标声明，不参与 `resource.operation` 调用。
- `workflow.operations` 是可选扩展；只声明 `processKey` 也可以表示一个工作流资源。
- `resources` 只定义协议级能力，不包含缓存、响应快照或设计器临时状态。

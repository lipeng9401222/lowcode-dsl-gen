# 场景示例

| 场景 | 参考文档 | 适用情况 |
|------|----------|----------|
| 列表页 | [`列表示例.md`](列表示例.md) | 查询区、表格、分页、工具栏、行操作 |
| 表单页 | [`表单示例.md`](表单示例.md) | 新增、编辑、详情、工作流办理表单 |

## 1. 列表页

适用需求：

- 用户说“列表、查询、表格、台账、管理页”。
- 页面需要展示多条业务数据。
- 可能包含查询条件、工具栏按钮、行按钮、分页、排序、选择。

推荐结构：

- 一个 `collection` 模型承载表格数据。
- 可选一个 `record` 模型承载查询条件。
- 根视图通常包含查询区、工具栏、表格和分页。
- 有真实接口时生成 `endpoint` 资源和刷新 action。
- 无真实接口且平台标准列表可满足时，不虚构 endpoint。

常见事件：

| 触发点 | 事件 | action |
|--------|------|--------|
| 页面加载 | `events.load` | 刷新列表 |
| 查询按钮 | `click` | 刷新列表 |
| 重置按钮 | `click` | 重置查询模型并刷新 |
| 分页变化 | `pageChange` | 刷新列表 |
| 排序变化 | `sortChange` | 刷新列表 |
| 行按钮 | `click` | 打开详情、编辑、删除确认或状态变更 |

生成注意：

- 标准查询、详情、新增、编辑、删除不主动生成 `event`。
- 状态变更按钮如“提交、撤回、作废、审批通过”如果伴随日志/消息/联动，可在用户明确要求时引用 `event`。
- 表格列字段必须能回到模型字段；操作列使用 table 的结构化列定义。

## 2. 表单页

适用需求：

- 用户说“新增、编辑、详情、表单、申请单、办理页”。
- 页面围绕单条业务数据填写、查看或提交。

推荐结构：

- 一个 `record` 模型承载主表单。
- 字段控件通过 `model` 绑定。
- 下拉、树选、人员选择等值/文本控件使用 `model` + `textModel`。
- 保存、提交、返回等按钮通过 `actions` 承载行为。
- 没有接口时只生成页面结构，不假设保存逻辑。

常见事件：

| 触发点 | 事件 | action |
|--------|------|--------|
| 页面加载 | `load` | 加载详情或初始化默认值 |
| 保存按钮 | `click` / `submit` | 校验并保存 |
| 重置按钮 | `click` | 重置模型 |
| 取消/返回 | `click` | 路由返回或关闭 |
| 字段变化 | `change` | 联动赋值、刷新选项或显隐 |

## 3. 详情/浏览页

当前可参考 `表单示例.md` 生成，只是控件应转为只读或展示型组件。

规则：

- 仍使用 `record` 模型。
- 页面 `load` 可绑定详情加载 action。
- 不生成保存按钮。
- 工作流浏览页的 `pagetag` 必须和 `workflow` 浏览节点 `handleurl` 一致。

## 4. 工作流办理页

当前可参考 `表单示例.md`，但要额外关注和 workflow 的关系。

规则：

- 申请页、审批页、浏览页建议使用不同 `pagetag`，避免权限和按钮混杂。
- workflow 的 `handleurl` / `mobilehandleurl` 引用页面 `pagetag`。
- 审批通过、退回等流程按钮优先由 workflow 配置承载；页面只负责表单展示和字段编辑。
- 若页面按钮需要触发动作流，必须明确 `event` 资产已存在或在计划中生成。

## 待补示例

- 工作流申请页：被申请节点 `handleurl` 引用的表单页。
- 工作流审批页：审批按钮、意见字段、流程动作绑定。
- 模块菜单列表页：与 `module` 菜单 URL 联动的列表入口。

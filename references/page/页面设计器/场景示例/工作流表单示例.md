# 工作流表单页场景示例

> 适用于“工作流申请页、办理页、审批页、流程浏览页”。普通表单完整组件结构参考 [`../../设计器 Schema 规范定义/表单示例.md`](../../设计器%20Schema%20规范定义/表单示例.md)，工作流表单完整 JSON 参考 [`../../设计器 Schema 规范定义/工作流表单示例.md`](../../设计器%20Schema%20规范定义/工作流表单示例.md)。

## 生成前提

- 表单页使用 `models[].type=record`。
- 表单页必须有真实顶层 `pageId`，供工作流列表新增、修改、查看入口引用。
- 表单页必须有稳定 `pagetag`，供 workflow 的 `handleurl` 或 `mobilehandleurl` 引用。
- workflow 资产中的申请、审批、浏览节点应明确引用哪个 `pagetag`。
- 字段、必填、只读、附件材料等必须来自需求、MIS、流程材料配置或已有页面。

## 使用脚本一键生成

当前 `add_page.py` 若只支持普通表单骨架，应先生成普通表单，再按本场景规则替换普通保存按钮并补充工作流组件。

```bash
python3 scripts/add_page.py \
  --app-root /path/to/<apptag> \
  --type form \
  --title 立项管理表单工作流 \
  --pagetag projectinfo_workflow_form \
  --page-id page_1780888036004 \
  --model projectinfo \
  --sql-table projectinfo \
  --fields-json '[{"name":"projectname","label":"项目名称","type":"string","required":true}]'
```

## 组件要点

- 顶部工具条使用 `toolbar`，按钮区放 `workflow-button`。
- 顶部工具条 actions 区放 `workflow-right`。
- 表单主体仍使用 `collapse -> collapse-item -> form-layout -> form-item -> control`。
- 流程历史组件 `workflow-history` 放在表单主体之后。
- 可按真实导出增加 `scrollbar` 和右侧 `anchor`，用于长表单滚动和分组定位。
- 普通底部 `save-button` 不应和 `workflow-button` 同时作为主提交入口。

## pageId 与 pagetag

工作流表单通常同时需要 `pageId` 和 `pagetag`：

```json
{
  "pageId": "page_1780888036004",
  "pagetag": "projectinfo_workflow_form",
  "kind": "page",
  "title": "立项管理表单工作流"
}
```

用途不同：

| 字段 | 用途 |
|------|------|
| `pageId` | 页面设计器运行时弹窗 URL 使用，列表新增、修改、查看按钮引用 |
| `pagetag` | workflow 的 `handleurl`、`mobilehandleurl` 引用 |

## 工作流组件

工作流办理页至少应包含：

| 组件 | 作用 |
|------|------|
| `workflow-button` | 保存、提交、审批、退回、流转等流程动作 |
| `workflow-right` | 工作流右侧辅助区 |
| `workflow-history` | 审批历史和流转记录 |

## 附件上传

附件上传按需生成：

- 需求明确有附件、材料、上传证明时生成。
- workflow 材料配置要求附件时生成。
- 只有普通字段时不要默认生成。

## 禁止事项

- 不在工作流表单中默认生成普通底部 `save-button` 作为主提交入口。
- 不因为是工作流表单就默认生成附件上传。
- 不把 `pageId` 和 `pagetag` 混用。
- 不主动生成 event 来承载标准保存、提交、查看；流程流转优先由 workflow 资产承载。

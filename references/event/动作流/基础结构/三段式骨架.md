# 标准三段式动作流骨架

## 概述

99% 的接口动作流都遵循"三段式"骨架：

```
[Webhook触发] → [初始化上下文] → [接口鉴权] → [业务方法] → [构建结果集] → [结束]
```

这是动作流的推荐架构。参考本工程现有的 `purchaseproject_getDataGridModel.event.yml`，即为标准三段式。

> ⚠️ **极复杂结构**：动作流 yml 由 `app`（元信息）+ `workflow.graph.nodes`（节点）+ `workflow.graph.edges`（边）三块构成，节点字段层层嵌套、坐标位置都在 yml 里，**不适合纯手写**，主要由设计器画图导出。本 skill 主要支持：1) 创建标准三段式骨架 2) 修改节点的 formData/business action 等关键参数。

## 模板填充策略（避免手写超长 yml）

由于动作流 yml 极复杂，**强烈推荐用模板替换关键变量**而非从零手写。

### 标准三段式骨架模板

模板文件位于 `assets/templates/event.yml`，预留以下占位符，由 `add_event.py` 脚本替换：

| 占位符 | 含义 | 示例值 |
|--------|------|--------|
| `{{EVENT_NAME}}` | 动作流名称 | `获取采购立项列表` |
| `{{EVENT_SIGN}}` | 接口标识 | `getDataGridModel` |
| `{{EVENT_ID}}` | 整体 UUID（app.id 与 app.rowguid） | 自动生成 |
| `{{CREATED_AT}}` | 创建时间（秒级时间戳） | 自动生成 |
| `{{NODE_START_ID}}` | 起始节点 ID | 自动生成（毫秒时间戳） |
| `{{NODE_INIT_ID}}` | 初始化节点 ID | 自动生成 |
| `{{NODE_AUTH_ID}}` | 鉴权节点 ID | 自动生成 |
| `{{NODE_BIZ_ID}}` | 业务节点 ID | 自动生成 |
| `{{NODE_RESULT_ID}}` | 结果节点 ID | 自动生成 |
| `{{NODE_END_ID}}` | 结束节点 ID | 自动生成 |
| `{{APPTAG}}` | apptag | `purchaseproject` |
| `{{BIZ_ACTION_NAME}}` | 业务方法名 | `PurchaseProjectListRestService_getDataGridModel` |
| `{{BIZ_TITLE}}` | 业务节点标题 | `获取列表分页数据` |
| `{{CONTEXT_CLASS}}` | 上下文 Java 类全限定名 | `com.epoint.xxx.PurchaseProjectContext` |
| `{{WEBHOOK_URL}}` | Webhook URL | `http://localhost:8080/EpointFrame/rest/dynamicapi/getDataGridModel` |
| `{{FORMULA_VALUE_ID}}` | formula.value[].id 中的 UUID 部分 | 自动生成 |
| `{{FORMULA_RESULT_ID}}` | 结果集 formula.value[].id 中的 UUID 部分 | 自动生成 |
| `{{FORMULA_END_ID}}` | 结束节点 formula.value[].id 中的 UUID 部分 | 自动生成 |

### 节点 ID 生成规则

每个节点的 `id` 是"毫秒时间戳"风格的字符串，如 `1770776813956`。生成方式：

```python
import time
node_id = str(int(time.time() * 1000))
# 多个节点用递增间隔
node_ids = [str(int(time.time() * 1000) + i * 1000) for i in range(6)]
```

### Edge ID 生成规则

```
<source_id>-source-<target_id>-target
```

直接由前后两节点 ID 拼接。

## 对话脚本（创建标准三段式动作流）

### 第一步：基本信息

```
你要创建什么动作流？
- 动作流名称（中文，如 获取采购立项列表）：
- 接口标识 sign（如 getDataGridModel，通常与 controller 方法同名）：
- 描述（可选）：
```

### 第二步：业务核心

```
- 业务方法 action（如 PurchaseProjectListRestService_getDataGridModel）：
- 上下文 Java 类全名（如 com.epoint.xxx.PurchaseProjectContext）：
- variableClassKey（一般等于 apptag，如 purchaseproject）：
```

### 第三步：请求体结构（可选，复杂可跳过用默认）

```
请求体结构是？
1. 标准列表分页（包含 conditions/sortOrders/pageSize/pageIndex）[默认推荐]
2. 标准详情（包含 id 字段）
3. 自定义（请描述请求字段）
```

### 第四步：响应结构

```
响应取值表达式？[默认: RuleParamUtil.parserParam(<上一节点ID>Context, "data.result")]
```

### 第五步：更新计划与追溯 + 明确批准

```
我准备创建这个动作流：

- 路径：<app-root>/event/<sign>.event.yml
- 内容：6 节点三段式骨架（webhook → 初始化 → 鉴权 → 业务 → 结果 → 结束）
- 业务方法：<action>
- 上下文类：<context class>

请回复"批准创建"后我再写文件；也可以直接说要调整哪一项。
```

每轮触发方式、业务方法、上下文类、鉴权和出参问题都要写入计划文档的 `对话确认记录`；用户回复后回填确认结果，再决定是否进入批准。

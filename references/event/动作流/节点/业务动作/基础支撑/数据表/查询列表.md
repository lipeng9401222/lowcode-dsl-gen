# 数据表查询列表节点 (biz-action - misfindlistaction)

## 概述

数据表查询列表节点用于批量查询数据表记录，支持分页、排序和复杂条件查询。返回多条记录的列表数据。

## formData 配置

### 1. sqlTableName（表名）

指定要查询的数据表名称。

```yaml
- name: "sqlTableName"
  valueMode: "fixed"
  value: "FRAME_USER"        # 数据表名
```

### 2. conditions（查询条件）

定义查询的条件表达式。

```yaml
- name: "conditions"
  valueMode: "fixed"
  value:
    type: "simple"           # 条件类型：simple / sql
    conditionData: {}        # 简单条件配置
    sqlData: {}              # SQL 条件配置
```

### 3. queryFields(查询字段)

指定需要查询的字段列表。

```yaml
- name: "queryFields"
  valueMode: "fixed"
  value:
  - name: "userguid"
    tableId: 2099122154      # ⚠️ 必填:数据表ID,从 query_table 返回的 tableID 获取
    label: "userguid"
    type: "string"
```

⚠️ **重要提示**:
- `tableId` 字段为**必填项**,缺少会导致导入报错
- 必须使用 `mcp_LowcodeMCP_query_table` 工具查询表结构,从返回结果的 `tableID` 字段获取
- 所有 `queryFields` 数组中的每一项都必须包含 `tableId` 字段

### 4. pageIndex（页码）

分页查询的页码（从1开始）。

```yaml
- name: "pageIndex"
  valueMode: "fixed"
  value: 1                   # 第1页，0表示不分页
```

### 5. pageSize（每页大小）

每页显示的记录数。

```yaml
- name: "pageSize"
  valueMode: "fixed"
  value: 10                  # 每页10条，0表示查询全部
```

## 条件查询配置

### 简单条件模式 (type: "simple")

使用可视化方式配置查询条件。

```yaml
conditionData:
  name: "and"                 # 逻辑连接符：and / or
  type: "group"               # 类型：group
  items:                      # 条件项数组
  - opt: "=="                 # 操作符
    left:                     # 左侧操作数
    - dataType: "string"
      type: "tableField"      # 表字段
      value: "userguid"       # 字段名
      label: "userguid"
    right:                    # 右侧操作数
    - dataType: "string"
      type: "field"           # 引用其他节点输出
      value: "s{节点ID}Context.data.body.userguid"
      label: "Webhook触发.Body.userguid"
```

#### 操作符 (opt)

| 操作符 | 说明 | 示例 |
|--------|------|------|
| `==` | 等于 | `userguid == "123"` |
| `!=` | 不等于 | `status != "deleted"` |
| `>` | 大于 | `age > 18` |
| `<` | 小于 | `amount < 1000` |
| `>=` | 大于等于 | `score >= 60` |
| `<=` | 小于等于 | `count <= 100` |
| `like` | 模糊匹配 | `name like "%张%"` |
| `in` | 在列表中 | `status in ("active", "pending")` |
| `not in` | 不在列表中 | `status not in ("deleted")` |
| `isNull` | 为空 | `remark isNull` |
| `isNotNull` | 不为空 | `email isNotNull` |
| `isBlank` | 为空或空字符串 | `remark isBlank` |
| `isNotBlank` | 不为空且非空字符串 | `email isNotBlank` |

#### 左侧操作数 (left)

通常是数据表字段。

```yaml
left:
- dataType: "string"          # 数据类型
  type: "tableField"          # 类型：表字段
  value: "userguid"           # 字段名
  label: "userguid"           # 显示标签
  formula: {}                 # 公式（可选）
```

#### 右侧操作数 (right)

可以是固定值或变量引用。

**固定值**：
```yaml
right:
- dataType: "string"
  type: "fix"                 # 类型：固定值
  value: "active"             # 固定值
  label: "固定值"
```

**引用变量**：
```yaml
right:
- dataType: "string"
  type: "field"               # 类型：字段引用
  value: "s1767841926276Context.data.body.keyword"
  label: "Webhook触发.Body.keyword"
```

### SQL 条件模式 (type: "sql")

使用自定义 SQL WHERE 条件。

```yaml
sqlData:
  name: "customSQL"
  value:
  - "userguid IS NOT NULL"
  - "AND isenabled = 1"
```

## 查询字段配置

`queryFields` 数组定义需要查询的字段列表。

```yaml
queryFields:
- name: "userguid"            # 字段名
  tableId: 2099122154         # ⚠️ 必填:数据表ID(从 query_table 返回的 tableID 字段获取)
  label: "userguid"           # 显示标签
  type: "string"              # 数据类型
- name: "loginid"
  tableId: 2099122154         # ⚠️ 必填:每个字段都需要配置
  label: "loginid"
  type: "string"
- name: "displayname"
  tableId: 2099122154
  label: "displayname"
  type: "string"
- name: "age"
  tableId: 2099122154
  label: "age"
  type: "int"
- name: "createtime"
  tableId: 2099122154
  label: "createtime"
  type: "datetime"
```

### 获取 tableId 的步骤

1. **调用 MCP 工具查询表结构**:
   ```
   mcp_LowcodeMCP_query_table(query="FRAME_USER")
   ```

2. **从返回结果中提取 tableID**:
   ```json
   {
     "tableID": 2099122154,       // ← 使用这个值
     "tableName": "FRAME_USER",
     "columns": [...]
   }
   ```

3. **配置到 queryFields**:
   ```yaml
   queryFields:
   - name: "userguid"
     tableId: 2099122154         // ← 配置 tableID
     label: "userguid"
     type: "string"
   ```

### 字段类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `string` | 字符串 | "张三" |
| `int` | 整数 | 123 |
| `decimal` | 小数 | 45.67 |
| `datetime` | 日期时间 | "2024-01-08 10:30:00" |
| `boolean` | 布尔值 | true / false |
| `text` | 长文本 | 大量文字 |

## 与迭代节点结合

查询列表常用于迭代节点，对每条记录进行处理。

```yaml
# 查询列表节点
- title: "查询用户列表"
  id: "query_users_id"
  actionData:
    action: "misfindlistaction"
    formData:
    - name: "sqlTableName"
      value: "FRAME_USER"
    - name: "conditions"
      value:
        type: "simple"
        conditionData:
          name: "and"
          type: "group"
          items:
          - opt: "=="
            left:
            - type: "tableField"
              value: "isenabled"
            right:
            - type: "fix"
              value: "1"

# 迭代节点
- title: "循环处理用户"
  type: "iteration"
  iterator_selector: ["query_users_id", "FRAME_USER"]  # 迭代查询结果
  
  # 在迭代内处理每个用户
  - title: "处理单个用户"
    isInIteration: true
    iteration_id: "iteration_id"
    # ... 处理逻辑，使用 s{iteration_id}Context.data.item.xxx 引用用户字段
```

## 最佳实践

### 1. 条件配置

- **优先使用简单模式**：简单条件模式更易维护
- **索引字段查询**：使用有索引的字段作为查询条件，提高性能
- **避免模糊查询**：尽量使用精确匹配，减少 LIKE 查询
- **合理使用 isNotBlank**：查询有效数据时使用非空条件

### 2. 分页配置

- **大数据量必须分页**：超过1000条记录建议启用分页
- **合理设置每页大小**：建议 10-100 条，根据业务场景调整
- **考虑性能影响**：分页可以减少内存占用和网络传输
- **前端联动**：分页参数通常从前端传递

### 3. 字段选择

- **只查询需要的字段**：避免查询所有字段 `SELECT *`
- **避免大字段**：如非必要，不要查询 BLOB、TEXT 等大字段
- **合理选择字段类型**：确保字段类型与数据库一致

### 4. 性能优化

- **使用索引字段**：WHERE 条件和 ORDER BY 字段应有索引
- **限制返回数量**：使用分页或 LIMIT 限制结果集
- **避免复杂条件**：简化查询条件，减少数据库负担
- **缓存查询结果**：对不常变化的数据，可以缓存结果

### 5. 结果处理

- **检查结果格式**：查询结果是数组，可能为空数组
- **使用迭代节点**：批量处理使用迭代节点遍历结果
- **处理空结果**：使用条件判断节点检查结果是否为空
- **引用结果数据**：迭代内使用 `s{迭代ID}Context.data.item.{字段名}`

### 6. 错误处理

- **配置错误处理模式**：根据业务需求选择 terminated 或 continue
- **处理空结果**：查询无结果不是错误，返回空数组
- **记录查询失败**：在错误分支记录失败信息

### 7. 数据量控制

- **评估数据量**：了解表的数据规模
- **设置合理的查询范围**：使用时间范围、状态等条件缩小范围
- **分批处理**：超大数据量考虑分批查询处理

## 性能优化

1. **使用索引字段**：查询条件和排序字段使用有索引的字段
2. **限制查询字段**：只查询必要的字段
3. **启用分页**：大数据量查询必须使用分页
4. **优化查询条件**：简化条件，避免复杂的嵌套
5. **缓存查询结果**：对不常变化的数据，可以缓存查询结果
6. **并行查询**：多个独立查询可以使用并行节点

## 常见问题

### 0. 导入报错:queryFields 缺少 tableId

**错误现象**:DSL 导入时提示字段配置错误

**原因**:`queryFields` 数组中的字段缺少 `tableId` 配置

**解决方法**:
```yaml
# ❌ 错误配置(缺少 tableId)
queryFields:
- name: "userguid"
  label: "userguid"
  type: "string"

# ✅ 正确配置
queryFields:
- name: "userguid"
  tableId: 2099122154      # 必须添加 tableId
  label: "userguid"
  type: "string"
```

**获取 tableId**:
1. 调用 `mcp_LowcodeMCP_query_table(query="表名")` 查询表结构
2. 从返回结果的 `tableID` 字段获取数值
3. 配置到每个字段的 `tableId` 属性

### 1. 查询无结果

**原因**：
- 条件不匹配
- 表名错误
- 字段名错误
- 引用的变量值错误

**解决**：
- 检查条件配置是否正确
- 确认表名和字段名拼写
- 使用代码节点打印变量值调试
- 检查数据库中是否有符合条件的数据

### 2. 查询结果为空数组

**原因**：没有符合条件的记录

**解决**：
- 使用条件判断节点检查数组长度
- 提供友好的空结果提示
- 放宽查询条件

### 3. 分页不生效

**原因**：pageIndex 或 pageSize 配置错误

**解决**：
- 确保 pageIndex 和 pageSize 都大于 0
- pageIndex 从 1 开始
- 检查变量引用是否正确

### 4. 查询超时

**原因**：
- 数据量过大
- 查询条件未使用索引
- 查询了大字段

**解决**：
- 启用分页限制返回数量
- 在索引字段上添加查询条件
- 移除不必要的大字段
- 优化数据库索引

### 5. 迭代处理失败

**原因**：
- 查询结果为空数组
- iterator_selector 配置错误

**解决**：
- 在迭代前使用条件判断检查结果
- 确认 iterator_selector 指向正确的查询结果
- 格式：`["query_node_id", "TABLE_NAME"]`

### 6. 类型不匹配

**原因**：字段类型配置与实际不符

**解决**：
- 检查 queryFields 中的 type 配置
- 确保条件中的 dataType 正确
- 验证数据库字段类型

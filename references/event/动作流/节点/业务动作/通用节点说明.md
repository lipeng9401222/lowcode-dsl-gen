# 业务动作节点通用说明

## 概述

业务动作节点是动作流中的核心执行单元，用于执行具体的业务逻辑操作，如数据查询、数据操作、服务调用等。业务动作节点通常由提供者（如数据表、统一用户等）提供，具有标准化的配置结构和执行流程。

## 节点配置

业务动作节点的配置遵循统一的YAML格式，包含节点基本信息、动作数据、表单数据等。

```yaml
data:
  provider_type: "frame"              # 提供者类型
  error_handle_mode: "terminated"     # 错误处理模式
  paramSchemas: []                    # 参数模式
  icon: "./logo/icon-tool9.png"       # 节点图标
  start_node_id: ""                   # 开始节点ID
  type: "biz-action"                  # 节点类型，固定为"biz-action"
  title: "节点标题"                    # 节点标题
  output_selector: []                 # 输出选择器
  iterator_selector: []               # 迭代选择器
  actionData:                         # 动作数据
    panelPath: "面板路径"              # 面板路径
    action: "动作名称"                # 动作名称
    formData: []                      # 表单数据
  tool_description: "工具描述"        # 工具描述
  isAction: true                      # 是否为动作节点
  provider_name: "提供者名称"         # 提供者名称
  editurl: ""                         # 编辑URL
  selected: false                     # 是否选中
  parallel_nums: 10                   # 并行数量
  tool_name: "工具名称"               # 工具名称
  variables: []                       # 变量
  params: []                          # 参数
  actionContext: []                   # 动作上下文
  real_type: "frame"                  # 实际类型
  provider_id: "提供者ID"             # 提供者ID
  output_schema: []                   # 输出模式
  is_parallel: false                  # 是否并行
  is_team_authorization: true         # 是否团队授权
  desc: ""                            # 描述
  tool_label: "工具标签"              # 工具标签
```

## 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| type | string | 节点类型，固定为 "biz-action" |
| actionData.action | string | 动作名称，由具体的业务动作定义 |
| tool_name | string | 工具名称，由具体的业务动作定义 |
| provider_name | string | 提供者名称，如"数据表"、"统一用户"等 |
| error_handle_mode | string | 错误处理模式：terminated（终止流程） / continue（继续执行） |
| is_parallel | boolean | 是否并行执行 |
| parallel_nums | integer | 并行执行数量 |
| is_team_authorization | boolean | 是否需要团队授权 |

## formData 配置

formData是业务动作节点的核心配置，用于定义节点执行所需的参数和数据。不同的业务动作节点可能有不同的formData配置项，但通常包含以下通用结构：

```yaml
formData:
- name: "参数名称"      # 参数名称
  valueMode: "fixed"    # 值模式：fixed（固定值） / formula（公式）
  value: "参数值"       # 参数值（valueMode为fixed时）
  formula: {}           # 公式配置（valueMode为formula时）
```

### 值模式 (valueMode)

| 模式 | 说明 | 使用场景 |
|------|------|----------|
| `fixed` | 固定值 | 直接指定常量值 |
| `formula` | 公式 | 使用表达式计算值或引用变量 |

### 公式配置 (formula)

公式配置用于动态计算参数值或引用其他节点的输出结果：

```yaml
formula:
  expression: "表达式"         # 实际执行的表达式
  header: "头部声明"           # 变量声明或导入
  text: "显示文本"             # 界面显示的简化文本
  value: []                   # 值数组
```

常用公式函数：

| 函数 | 说明 | 示例 |
|------|------|------|
| `StringFunctionRule.getUUID()` | 生成 UUID | 用于主键字段 |
| `RuleParamUtil.parserParam()` | 解析参数 | 引用其他节点输出 |
| `DateFunctionRule.now()` | 获取当前时间 | 时间戳字段 |
| `DateFunctionRule.formatDate()` | 格式化日期 | 日期格式转换 |

## 参数映射与检查

### 概述

参数映射是业务动作节点配置中的重要环节，用于确保节点执行时能够正确获取所需的输入参数。当参数涉及动态公式计算值或从上下文中获取数据时，需要进行严格的字段映射配置检查。

### 映射规则

1. **字段名匹配要求**：从上下文中获取的参数必须与表单数据（formData）中的字段名（name属性）完全匹配。

2. **动态公式检查**：当使用公式模式（valueMode="formula"）时，系统会自动检查公式中引用的上下文变量是否与表单字段名对应。

3. **空值处理机制**：如果检查发现从上下文中获取的参数在表单数据中没有对应的字段名，系统会自动为该参数分配一个空的固定值属性。

4. **参数配置规范**：参数名称必须基于工具文档或示例中给出的数据，不得编造不存在的参数。参数映射必须确保英文字段名或中文字段描述与实际业务字段对应；对于无法直接映射或不需要映射的参数，应使用固定值模式（valueMode="fixed"）并将value设置为空字符串，由用户根据实际需求自行配置。

### 配置示例

```yaml
# 正确的参数映射配置
formData:
- name: "user_id"              # 字段名
  valueMode: "formula"         # 使用公式模式
  formula:
    expression: "RuleParamUtil.parserParam('s123Context.data.user_id')"  # 引用上下文变量，与字段名user_id匹配
    text: "引用用户ID"

# 错误的参数映射配置（会触发空值处理）
formData:
- name: "user_id"              # 字段名
  valueMode: "formula"         # 使用公式模式
  formula:
    expression: "RuleParamUtil.parserParam('s123Context.data.userIdentifier')"  # 引用的上下文变量与字段名不匹配
    text: "引用用户标识"
```

### 空值处理效果

当字段映射检查失败时，系统会自动将参数转换为固定值模式并设置为空：

```yaml
# 自动转换后的配置
formData:
- name: "user_id"              # 保持原字段名
  valueMode: "fixed"           # 转换为固定值模式
  value: ""                    # 设置为空值
```

## 结果输出

业务动作节点执行成功后，结果会存储在节点上下文中，供后续节点使用。

```yaml
# 格式
s{节点ID}Context.data.{结果键}

# 示例
s1767841950223Context.data.FRAME_USER
```

不同类型的业务动作节点可能有不同的输出结构：

- **查询类节点**：返回查询结果（对象或数组）
- **操作类节点**：返回操作结果（如影响行数、成功状态等）
- **服务调用类节点**：返回服务调用结果

## 配置示例

业务动作节点的配置示例通常根据具体的业务场景提供，包括：

1. **基础配置示例**：展示节点的基本配置方法
2. **变量引用示例**：展示如何引用其他节点的输出
3. **条件配置示例**：展示如何配置查询或操作条件
4. **迭代使用示例**：展示如何与迭代节点配合使用
5. **高级配置示例**：展示复杂业务场景的配置方法

## 最佳实践

### 1. 节点配置

- **明确节点用途**：每个节点应专注于单一功能
- **配置错误处理**：根据业务需求选择合适的错误处理模式
- **合理设置并行**：对独立的节点可以启用并行执行提高效率

### 2. 参数配置

- **优先使用公式**：尽量使用公式引用其他节点的输出，避免硬编码
- **验证参数完整性**：确保所有必填参数都已配置
- **注意数据类型**：确保参数值的类型与字段要求匹配
- **保持字段名一致性**：确保上下文中引用的变量名与表单字段名（name属性）完全匹配，避免映射错误
- **避免动态变量名**：尽量使用固定的字段名，减少动态生成变量名导致的映射错误
- **定期检查映射关系**：在复杂流程中定期验证参数映射关系的正确性

### 3. 变量引用

- **使用完整路径**：引用变量时使用完整的上下文路径
- **检查变量存在性**：在使用变量前先判断变量是否存在
- **避免循环引用**：避免节点间的循环依赖

### 4. 性能优化

- **限制返回数据**：只查询或返回必要的字段
- **使用索引字段**：查询条件使用有索引的字段
- **批量处理数据**：大量数据处理时使用批量操作
- **缓存重复查询**：对不常变化的数据可以缓存查询结果

### 5. 错误处理

- **配置错误模式**：根据业务重要性选择合适的错误处理模式
- **处理空结果**：对查询类节点，处理可能的空结果情况
- **记录错误信息**：在错误分支记录详细的错误信息

## 常见问题

### 1. 节点执行失败

**原因**：
- 参数配置错误
- 数据类型不匹配
- 权限不足
- 外部服务异常

**解决**：
- 检查参数配置是否正确
- 验证数据类型是否匹配
- 确认用户权限
- 检查外部服务状态

### 2. 变量引用错误

**原因**：
- 引用路径不正确
- 引用的变量不存在
- 变量类型不匹配

**解决**：
- 检查变量引用路径
- 在使用前判断变量是否存在
- 确保变量类型与期望一致

### 3. 性能问题

**原因**：
- 查询条件未使用索引
- 返回数据过多
- 大量数据循环处理

**解决**：
- 优化查询条件，使用索引字段
- 限制返回字段和记录数
- 使用批量操作替代循环处理

### 4. 数据一致性问题

**原因**：
- 并发操作导致数据冲突
- 错误处理模式设置不当
- 分布式事务问题

**解决**：
- 使用事务保证数据一致性
- 设置合适的错误处理模式
- 避免长时间锁定数据

### 5. 参数映射错误

**原因**：
- 上下文中引用的变量名与表单字段名不匹配
- 动态公式中使用了不存在的上下文变量
- 字段名大小写不一致导致匹配失败

**解决**：
- 确保上下文中引用的变量名与表单字段名（name属性）完全一致
- 检查动态公式中使用的上下文变量是否存在
- 注意字段名的大小写敏感性
- 利用系统自动空值处理机制作为容错手段


## 面板配置说明指引

### 概述

当通过 `query_action` 查询到的业务动作资产包含 `panelPath` 字段时，说明该资产有对应的面板配置文档。面板配置文档定义了该类节点的完整参数结构、action 名称和 formData 配置要求。

### 面板配置查询流程

1. 使用 `query_action` 查询业务动作，检查返回结果是否包含 `panelPath` 字段
2. 根据 `panelPath` 值，使用 `read_document` 查询对应的面板配置文档
3. 按照面板配置文档的要求生成完整的节点配置（包括正确的 `action`、`tool_name` 和 `formData` 参数）

### 面板配置文档路径映射

| panelPath 值 | 对应面板配置文档 | 说明 |
|-------------|-----------------|------|
| `frame-datasource/data-insert` | `动作流/节点/业务动作/基础支撑/数据表/面板配置说明.md` | 新增数据 |
| `frame-datasource/data-update` | `动作流/节点/业务动作/基础支撑/数据表/面板配置说明.md` | 更新数据 |
| `frame-datasource/data-delete` | `动作流/节点/业务动作/基础支撑/数据表/面板配置说明.md` | 删除数据 |
| `frame-datasource/data-query-list` | `动作流/节点/业务动作/基础支撑/数据表/面板配置说明.md` | 查询列表 |
| `frame-datasource/data-query-detail` | `动作流/节点/业务动作/基础支撑/数据表/面板配置说明.md` | 查询详情 |
| `common/biz-action` | 通用业务动作（无需额外面板配置） | 标准业务动作 |
| `frame-webhook/frame-webhook` | Webhook 触发节点 | 触发节点 |

### 使用规则

- ✅ **必须**：当资产包含 `panelPath` 时，查询对应面板配置文档获取完整参数列表
- ✅ **必须**：按照面板配置文档中的 `action` 和 `formData` 参数配置节点
- ✅ **必须**：`actionData.category` 固定为 `biz_action`（仅适用于业务动作节点）
- ❌ **禁止**：忽略 `panelPath` 直接使用 `query_action` 返回的参数结构（可能不完整）

### 示例

```yaml
# 当 query_action 返回 panelPath: "frame-datasource/data-query-list" 时
# 查询面板配置文档获取完整配置：
data:
  type: "biz-action"
  title: "查询列表"
  actionData:
    panelPath: "frame-datasource/data-query-list"
    action: "misfindlistaction"        # 来自面板配置文档
    formData:
    - name: "sqlTableName"             # ⚠️ 注意：是 sqlTableName 不是 tableName
      valueMode: "fixed"
      value: "CG_ProjectInfo"
    - name: "conditions"
      valueMode: "fixed"
      value: {}
    - name: "queryFields"
      valueMode: "fixed"
      value: []
    - name: "sortFields"
      valueMode: "fixed"
      value: []
    - name: "pageInfo"
      valueMode: "fixed"
      value: {}
    category: "biz_action"
    panelType: ""
```

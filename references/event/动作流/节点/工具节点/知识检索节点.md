# 知识检索节点 (knowledge-retrieval)

## 概述

知识检索节点用于从配置的知识库中检索相关文档片段，为 LLM 节点提供上下文信息。支持单数据集和多数据集检索，可配置检索数量、重排序等参数。

## 节点配置

```yaml
data:
  dataset_ids: []                        # 数据集ID列表
  bizfilter: []                          # 业务过滤条件
  retrieval_mode: "multiple"             # 检索模式
  is_konwledge_formula: "固定值"          # 知识库公式类型
  query_variable_selector: []            # 查询变量选择器
  multiple_retrieval_config:             # 多数据集检索配置
    top_n: 4                             # 返回的最终结果数量
    top_k: 4                             # 每个数据集检索的数量
    reranking_enable: false              # 是否启用重排序
  is_model_formula: "固定值"              # 模型公式类型
  type: "knowledge-retrieval"            # 节点类型
  title: "知识检索"                       # 节点标题
  selected: false                        # 是否选中
  desc: ""                               # 节点描述
```

## 字段说明

### 核心字段

| 字段 | 类型 | 说明 |
|------|------|------|
| type | string | 节点类型，固定为 "knowledge-retrieval" |
| title | string | 节点显示标题 |
| dataset_ids | array | 要检索的数据集ID列表 |
| retrieval_mode | string | 检索模式：single（单数据集）/ multiple（多数据集） |
| query_variable_selector | array | 查询文本的变量选择器 |

### 检索模式 (retrieval_mode)

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `single` | 单数据集检索 | 只需要从一个知识库检索 |
| `multiple` | 多数据集检索 | 需要从多个知识库检索并合并结果 |

### 多数据集检索配置 (multiple_retrieval_config)

```yaml
multiple_retrieval_config:
  top_n: 4                   # 返回的最终结果数量
  top_k: 4                   # 每个数据集检索的数量
  reranking_enable: false    # 是否启用重排序
```

| 字段 | 类型 | 说明 | 推荐值 |
|------|------|------|--------|
| top_k | number | 从每个数据集检索的文档数量 | 3-10 |
| top_n | number | 最终返回的文档数量 | 3-5 |
| reranking_enable | boolean | 是否启用重排序（提高准确性） | true（推荐） |

**参数说明**：
- **top_k**：从每个数据集独立检索 top_k 个结果
- **top_n**：对所有检索结果进行合并和（可选）重排序后，返回 top_n 个最相关的结果
- **reranking**：使用重排序模型对检索结果进行重新排序，提高相关性

### 业务过滤 (bizfilter)

对检索结果进行业务维度的过滤：

```yaml
bizfilter:
- field: "category"          # 字段名
  operator: "equals"         # 操作符
  value: "产品文档"           # 过滤值
- field: "status"
  operator: "in"
  value: ["published", "reviewed"]
```

**支持的操作符**：
- `equals`：等于
- `not_equals`：不等于
- `in`：在列表中
- `not_in`：不在列表中
- `contains`：包含
- `greater_than`：大于
- `less_than`：小于

### 查询变量选择器 (query_variable_selector)

指定查询文本的来源：

```yaml
# 引用 Webhook 输入
query_variable_selector:
- "{{#1768549737010.body.question#}}"

# 引用代码节点输出
query_variable_selector:
- "{{#1768549742985.query#}}"
```

### 数据集配置 (dataset_ids)

配置要检索的知识库数据集：

```yaml
dataset_ids:
- "dataset-uuid-001"         # 数据集1
- "dataset-uuid-002"         # 数据集2
- "dataset-uuid-003"         # 数据集3
```

## 配置示例

### 场景一：单知识库问答

```yaml
data:
  type: "knowledge-retrieval"
  title: "检索产品文档"
  retrieval_mode: "single"
  dataset_ids:
  - "product-docs-dataset-id"
  query_variable_selector:
  - "{{#1768549737010.body.question#}}"
  multiple_retrieval_config:
    top_n: 3                    # 返回3个最相关片段
    top_k: 3
    reranking_enable: true      # 启用重排序提高准确性
```

**工作流示例**：

```
Webhook触发(接收问题)
    ↓
知识检索(检索产品文档)
    ↓
LLM(基于文档回答问题)
    ↓
结束(返回答案)
```

### 场景二：多知识库检索

```yaml
data:
  type: "knowledge-retrieval"
  title: "检索多个知识库"
  retrieval_mode: "multiple"
  dataset_ids:
  - "product-docs-id"          # 产品文档
  - "api-docs-id"              # API文档
  - "faq-docs-id"              # FAQ文档
  query_variable_selector:
  - "{{#1768549737010.body.question#}}"
  multiple_retrieval_config:
    top_k: 5                    # 从每个库检索5个
    top_n: 3                    # 最终返回3个最相关的
    reranking_enable: true      # 启用重排序
```

**说明**：
- 从3个数据集各检索5个文档（共15个）
- 重排序后返回最相关的3个文档
- 适合需要综合多个知识源的场景

### 场景三：带业务过滤的检索

```yaml
data:
  type: "knowledge-retrieval"
  title: "检索已发布文档"
  retrieval_mode: "multiple"
  dataset_ids:
  - "knowledge-base-id"
  query_variable_selector:
  - "{{#1768549737010.body.question#}}"
  bizfilter:                    # 业务过滤
  - field: "status"
    operator: "equals"
    value: "published"          # 只检索已发布的文档
  - field: "language"
    operator: "equals"
    value: "zh-CN"              # 只检索中文文档
  multiple_retrieval_config:
    top_n: 4
    top_k: 4
    reranking_enable: true
```

## 最佳实践

### 1. 检索参数调优

| 场景 | top_k | top_n | reranking |
|------|-------|-------|-----------|
| 精确问答 | 3-5 | 2-3 | ✅ 启用 |
| 综合分析 | 5-10 | 5-8 | ✅ 启用 |
| 快速检索 | 3 | 3 | ❌ 禁用 |
| 多知识库 | 5-8 | 3-5 | ✅ 必须启用 |

### 2. 知识库组织

- **主题分类**：按业务领域或主题创建不同的数据集
- **粒度控制**：文档分块大小适中（200-500字）
- **元数据丰富**：添加标题、分类、标签等元数据
- **定期更新**：保持知识库内容的时效性

### 3. 查询优化

```groovy
// 使用代码节点优化查询
def main(user_question) {
    // 1. 清理查询文本
    def cleaned = user_question.trim()
    
    // 2. 提取关键词
    def keywords = extractKeywords(cleaned)
    
    // 3. 构建增强查询
    def enhanced = """
    问题：${cleaned}
    关键词：${keywords.join(', ')}
    """
    
    return [query: enhanced]
}
```

### 4. 结果过滤

```groovy
// 对检索结果进行二次过滤
def main(kb_result) {
    def parsed = parseKBResult(kb_result)
    
    // 过滤低相关性结果
    def filtered = parsed.findAll { 
        it.score > 0.7  // 只保留相关性 > 0.7 的结果
    }
    
    // 按相关性排序
    def sorted = filtered.sort { -it.score }
    
    // 提取纯文本内容
    def content = sorted.collect { it.content }.join("\n\n")
    
    return [filtered_result: content]
}
```

### 5. 多轮检索

```
第一次检索
    ↓
LLM判断是否需要更多信息
    ↓ 是
第二次检索（调整查询或数据集）
    ↓
LLM综合回答
```

### 6. 性能优化

- **合理设置 top_k**：不要设置过大（一般不超过10）
- **选择性启用重排序**：重排序会增加响应时间但提高准确性
- **缓存热门查询**：对常见问题缓存检索结果
- **异步检索**：多个数据集可以并行检索

## 常见问题

### 1. 检索结果不相关

**原因**：
- 查询文本不够明确
- 知识库内容质量问题
- 未启用重排序

**解决**：
```yaml
# 1. 优化查询文本
query_variable_selector:
- "{{#代码节点ID.enhanced_query#}}"  # 使用增强后的查询

# 2. 启用重排序
multiple_retrieval_config:
  reranking_enable: true              # 启用重排序

# 3. 调整检索数量
multiple_retrieval_config:
  top_k: 8                            # 增加候选数量
  top_n: 3                            # 保持返回数量
```

### 2. 检索结果过少

**原因**：
- 知识库内容不足
- 过滤条件过严
- top_n 设置过小

**解决**：
```yaml
# 增加检索数量
multiple_retrieval_config:
  top_n: 5                  # 增加到5个

# 检查业务过滤
bizfilter: []               # 暂时移除过滤测试

# 增加数据集
dataset_ids:
- "dataset-1"
- "dataset-2"               # 添加更多数据集
```

### 3. 响应时间过长

**原因**：
- 检索数量过多
- 启用了重排序
- 数据集过大

**解决**：
```yaml
# 优化配置
multiple_retrieval_config:
  top_k: 3                  # 减少检索数量
  top_n: 3
  reranking_enable: false   # 禁用重排序（如果时间敏感）

# 优化知识库
# - 拆分大数据集
# - 优化索引
# - 使用更快的向量数据库
```

### 4. 多知识库结果不均衡

**原因**：某些知识库的内容相关性明显更高

**解决**：
```yaml
# 方案1：启用重排序
multiple_retrieval_config:
  reranking_enable: true    # 重排序会平衡各数据集的结果

# 方案2：分类检索
# 先判断问题类型，再选择对应的数据集

# 方案3：多次检索
# 对不同数据集进行独立检索，然后合并结果
```

### 5. 检索内容过时

**原因**：知识库未及时更新

**解决**：
- 建立知识库更新机制
- 在元数据中添加时间戳
- 使用业务过滤排除过时内容

```yaml
bizfilter:
- field: "update_time"
  operator: "greater_than"
  value: "2024-01-01"       # 只检索2024年后的内容
```

## 与 LLM 节点配合

### 标准 RAG 模式

```yaml
# 1. 知识检索节点
data:
  type: "knowledge-retrieval"
  title: "检索知识"
  dataset_ids: ["kb-id"]
  query_variable_selector:
  - "{{#1768549737010.body.question#}}"
  multiple_retrieval_config:
    top_n: 3
    top_k: 3
    reranking_enable: true

# 2. LLM节点
---
data:
  type: "llm"
  title: "回答问题"
  model:
    name: "gpt-4"
    completion_params:
      temperature: 0.3        # 低温度确保答案准确
  prompt_template:
  - role: "system"
    text: "你是一个专业的知识助手，只基于提供的知识库内容回答问题。"
  - role: "user"
    text: |
      知识库内容：
      {{#知识检索节点ID.result#}}
      
      用户问题：
      {{#1768549737010.body.question#}}
      
      回答要求：
      1. 仅根据知识库内容回答
      2. 如果知识库中没有相关信息，明确说明
      3. 引用具体的知识库内容
```

### 多轮对话 RAG

```yaml

```

## 注意事项

1. **数据集ID有效性**：确保配置的数据集ID存在且有效
2. **查询文本质量**：查询文本的质量直接影响检索效果
3. **top_k 和 top_n 关系**：top_n 应该 ≤ top_k × 数据集数量
4. **重排序成本**：重排序会增加响应时间，权衡准确性和速度
5. **知识库质量**：定期审核和更新知识库内容
6. **元数据利用**：充分利用元数据进行过滤和排序

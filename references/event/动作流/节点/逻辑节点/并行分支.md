# 并行分支节点 (parallelbranch)

## 概述

并行分支节点用于将流程分成多个并行执行的分支,每个分支独立执行,互不影响。适用于需要同时执行多个独立任务的场景。

## 节点配置

```yaml
data:
  variables: []                    # 变量列表
  is_async: false                  # 是否异步执行
  type: "parallelbranch"           # 节点类型
  title: "并行分支"                 # 节点标题
  selected: false
  desc: ""                         # 节点描述
targetPosition: "left"             # 目标位置
width: 244                         # 节点宽度
sourcePosition: "right"            # 源位置
id: "1768549901364"               # 节点ID
type: "custom"                     # 自定义类型
resultType: 1                      # 结果类型
height: 54                         # 节点高度
```

## 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| type | string | 节点类型,固定为 "parallelbranch" |
| title | string | 节点显示标题 |
| is_async | boolean | 是否异步执行 |
| variables | array | 变量列表 |
| resultType | number | 结果类型,1表示标准结果 |

## 异步执行配置 (is_async)

控制分支是否异步执行。

| 值 | 说明 | 适用场景 |
|----|------|----------|
| `false` | 同步并行执行(默认) | 所有分支执行完毕后才继续后续流程 |
| `true` | 异步并行执行 | 启动所有分支后立即继续后续流程,不等待分支完成 |

```yaml
# 同步并行(等待所有分支完成)
is_async: false

# 异步并行(启动分支后立即继续)
is_async: true
```

## 连接配置

并行分支节点可以连接多个后续节点,每个连接代表一个独立的并行分支。

```yaml
edges:
# 分支1
- sourceHandle: "source"
  source: "1768549901364"
  target: "branch1_node_id"
  targetHandle: "target"

# 分支2
- sourceHandle: "source"
  source: "1768549901364"
  target: "branch2_node_id"
  targetHandle: "target"

# 分支3
- sourceHandle: "source"
  source: "1768549901364"
  target: "branch3_node_id"
  targetHandle: "target"
```

## 实际应用场景

### 场景一:多数据源同步

```
接收同步请求
    ↓
并行分支
    ↓
    ├─→ 分支1:同步用户数据
    ├─→ 分支2:同步部门数据
    ├─→ 分支3:同步角色数据
    └─→ 分支4:同步权限数据
    ↓
汇总结果
    ↓
返回同步状态
```

### 场景二:多渠道消息推送

```
触发消息推送
    ↓
并行分支(异步)
    ↓
    ├─→ 发送邮件
    ├─→ 发送短信
    ├─→ 发送APP推送
    └─→ 发送站内消息
    ↓
记录推送状态
```

### 场景三:数据处理和审计并行

```
接收业务请求
    ↓
并行分支
    ↓
    ├─→ 主流程:业务处理
    └─→ 副流程:审计日志
    ↓
返回处理结果
```

### 场景四:多接口并行调用

```
API网关接收请求
    ↓
并行分支
    ↓
    ├─→ 调用服务A接口
    ├─→ 调用服务B接口
    └─→ 调用服务C接口
    ↓
聚合多个接口结果
    ↓
返回统一响应
```

## 同步并行 vs 异步并行

### 同步并行 (is_async: false)

**执行流程**:
```
上游节点
    ↓
并行分支(同步)
    ↓
    ├─→ 分支1 ────┐
    ├─→ 分支2 ────┤ 等待所有分支完成
    └─→ 分支3 ────┘
    ↓
下游节点(所有分支完成后才执行)
```

**特点**:
- 等待所有分支执行完毕
- 可以获取所有分支的执行结果
- 适合需要汇总多个分支结果的场景
- 总执行时间=最慢分支的执行时间

**适用场景**:
- 多数据源查询后需要聚合结果
- 多个接口调用后需要整合数据
- 并行处理后需要统一结果

### 异步并行 (is_async: true)

**执行流程**:
```
上游节点
    ↓
并行分支(异步)
    ↓
    ├─→ 分支1(后台执行)
    ├─→ 分支2(后台执行)
    └─→ 分支3(后台执行)
    ↓
下游节点(立即执行,不等待分支)
```

**特点**:
- 启动分支后立即继续
- 无法获取分支的执行结果
- 适合"发送即忘记"的场景
- 提高主流程响应速度

**适用场景**:
- 消息通知(邮件、短信)
- 日志记录
- 数据备份
- 异步任务触发

## 结果汇总

### 同步并行结果聚合

使用代码节点汇总多个分支的执行结果:

```groovy
def main(branch1Result, branch2Result, branch3Result) {
    return [
        branch1: [
            success: branch1Result?.success ?: false,
            data: branch1Result?.data
        ],
        branch2: [
            success: branch2Result?.success ?: false,
            data: branch2Result?.data
        ],
        branch3: [
            success: branch3Result?.success ?: false,
            data: branch3Result?.data
        ],
        allSuccess: (branch1Result?.success && 
                     branch2Result?.success && 
                     branch3Result?.success)
    ]
}
```

### 异步并行状态跟踪

异步并行无法直接获取结果,可通过以下方式跟踪:

**方式1:数据库记录**
```
并行分支(异步)
    ↓
    ├─→ 分支1 → 记录状态到数据库
    ├─→ 分支2 → 记录状态到数据库
    └─→ 分支3 → 记录状态到数据库

单独流程:查询数据库获取各分支状态
```

**方式2:回调接口**
```
并行分支(异步)
    ↓
    ├─→ 分支1 → 完成后调用回调接口
    ├─→ 分支2 → 完成后调用回调接口
    └─→ 分支3 → 完成后调用回调接口
```

## 最佳实践

### 1. 分支设计原则

**独立性**:每个分支应该相互独立,没有依赖关系
```yaml
# ✅ 正确:分支独立
并行分支
├─→ 查询用户表
├─→ 查询部门表
└─→ 查询角色表

# ❌ 错误:分支有依赖
并行分支
├─→ 查询用户表
└─→ 使用用户ID查询订单表(依赖第一个分支)
```

**无序性**:分支执行顺序不确定,不应依赖执行顺序
```yaml
# ✅ 正确:顺序无关
并行分支
├─→ 发送邮件
└─→ 发送短信

# ❌ 错误:依赖顺序
并行分支
├─→ 创建订单(需要先执行)
└─→ 创建订单明细(依赖订单ID)
```

### 2. 同步 vs 异步选择

**选择同步并行的情况**:
- 需要汇总所有分支的结果
- 后续流程依赖分支执行结果
- 需要确保所有分支执行完成
- 关键业务流程

**选择异步并行的情况**:
- 不关心分支执行结果
- 提高主流程响应速度
- 通知类操作
- 非关键业务流程

### 3. 错误处理

**同步并行**:
```yaml
# 每个分支都应有错误处理
并行分支
    ↓
    ├─→ 分支1
    │   ├─ 成功 → 返回结果
    │   └─ 失败 → 返回错误信息
    ├─→ 分支2
    │   ├─ 成功 → 返回结果
    │   └─ 失败 → 返回错误信息
    └─→ 分支3
        ├─ 成功 → 返回结果
        └─ 失败 → 返回错误信息
    ↓
检查所有分支结果
    ↓
    ├─ 全部成功 → 继续流程
    ├─ 部分失败 → 记录错误,继续流程
    └─ 全部失败 → 终止流程
```

**异步并行**:
```yaml
# 分支内部处理错误
并行分支(异步)
    ↓
    ├─→ 分支1
    │   └─ 内部捕获异常并记录
    ├─→ 分支2
    │   └─ 内部捕获异常并记录
    └─→ 分支3
        └─ 内部捕获异常并记录
```

### 4. 性能优化

**控制分支数量**:
- 建议分支数量: 2-5 个
- 最多分支数量: 不超过 10 个
- 过多分支会增加系统负担

**优化分支执行时间**:
```
理想情况:各分支执行时间接近
分支1: ████████ (8秒)
分支2: ████████ (8秒)
分支3: ████████ (8秒)
总时间: 8秒

不理想情况:执行时间差距大
分支1: ██ (2秒)
分支2: ████ (4秒)
分支3: ████████████████ (16秒)
总时间: 16秒 (被最慢的分支拖累)
```

**优化建议**:
- 将耗时长的操作拆分为多个小操作
- 将快速操作和慢速操作分开处理
- 使用异步并行处理耗时长且不重要的操作

### 5. 监控和调试

**添加日志节点**:
```yaml
并行分支
    ↓
    ├─→ 分支1
    │   ├─ 记录开始时间
    │   ├─ 执行业务逻辑
    │   └─ 记录结束时间和结果
    ├─→ 分支2
    │   ├─ 记录开始时间
    │   ├─ 执行业务逻辑
    │   └─ 记录结束时间和结果
    └─→ 分支3
        ├─ 记录开始时间
        ├─ 执行业务逻辑
        └─ 记录结束时间和结果
```

**监控关键指标**:
- 各分支执行时间
- 分支成功率
- 总体执行时间
- 资源使用情况

## 常见问题

### 1. 分支之间如何共享数据?

**问题**:并行分支执行时,一个分支无法访问另一个分支的数据。

**解决方案**:

**方案1:使用上游节点的数据**
```yaml
# 所有分支都可以访问上游节点的输出
上游节点(ID: node_123)
    ↓
并行分支
    ├─→ 分支1:使用 node_123 的输出
    └─→ 分支2:使用 node_123 的输出
```

**方案2:通过数据库共享**
```yaml
并行分支
    ├─→ 分支1 → 写入数据库
    └─→ 分支2 → 读取数据库
```

### 2. 如何确保分支执行顺序?

**问题**:需要按特定顺序执行多个任务。

**解决**:不要使用并行分支,改用串行连接:
```yaml
# ❌ 错误:使用并行分支
并行分支
├─→ 任务1(需要先执行)
└─→ 任务2(需要后执行)

# ✅ 正确:使用串行连接
任务1
  ↓
任务2
```

### 3. 某个分支失败是否影响其他分支?

**同步并行**:
- 默认情况:一个分支失败不会影响其他分支继续执行
- 所有分支执行完后,汇总结果时可以看到哪些失败

**异步并行**:
- 各分支完全独立
- 一个分支失败不影响其他分支

### 4. 并行分支的性能提升有限?

**原因**:
- 分支执行时间差距大,总时间由最慢的分支决定
- 分支过多导致系统资源竞争
- 分支操作的服务器性能限制

**优化**:
- 平衡各分支的执行时间
- 减少分支数量
- 优化分支内部逻辑
- 考虑使用批处理代替多分支

```yaml
    检查结果,失败则继续循环
```

## 调试技巧

### 1. 输出分支执行信息

在每个分支添加代码节点:

```groovy
def main() {
    def startTime = System.currentTimeMillis()
    println "分支1开始执行: ${new Date()}"
    
    // 业务逻辑
    
    def endTime = System.currentTimeMillis()
    println "分支1执行完成,耗时: ${endTime - startTime}ms"
    
    return [
        branch: "branch1",
        startTime: startTime,
        endTime: endTime,
        duration: endTime - startTime
    ]
}
```

### 2. 模拟分支延迟

测试并行执行效果:

```groovy
def main() {
    println "分支1开始"
    Thread.sleep(3000)  // 模拟3秒延迟
    println "分支1完成"
    return [status: "success"]
}
```

### 3. 追踪分支执行状态

```groovy
def main() {
    def branchId = "branch_1"
    def status = "running"
    
    // 记录到数据库或日志
    logBranchStatus(branchId, status)
    
    try {
        // 业务逻辑
        status = "success"
    } catch (Exception e) {
        status = "failed"
        logError(branchId, e.message)
    } finally {
        logBranchStatus(branchId, status)
    }
    
    return [branch: branchId, status: status]
}
```

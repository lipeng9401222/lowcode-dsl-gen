# 循环节点 (loop)

## 概述

循环节点用于重复执行一组操作直到满足退出条件或达到最大循环次数。与迭代节点不同,循环节点支持条件退出,适用于需要根据执行结果动态决定是否继续循环的场景。

## 节点配置

```yaml
data:
  loop_variables: []               # 循环变量列表
  loop_count: 10                   # 最大循环次数
  logical_operator: "and"          # 逻辑运算符
  width: 388                       # 节点宽度
  start_node_id: "节点IDstart"     # 开始节点ID
  type: "loop"                     # 节点类型
  title: "循环"                     # 节点标题
  break_conditions: []             # 退出条件列表
  selected: false
  desc: ""                         # 节点描述
  height: 174                      # 节点高度
sourcePosition: "right"            # 源位置
type: "custom"                     # 自定义类型
targetPosition: "left"             # 目标位置
resultType: 2                      # 结果类型
zIndex: -999                       # 层级
```

## 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| type | string | 节点类型,固定为 "loop" |
| title | string | 节点显示标题 |
| loop_count | number | 最大循环次数(防止死循环) |
| break_conditions | array | 退出条件列表 |
| logical_operator | string | 逻辑运算符:and/or |
| loop_variables | array | 循环变量(用于记录状态) |
| start_node_id | string | 循环开始节点ID |

## 最大循环次数 (loop_count)

设置循环的最大执行次数,防止死循环。

```yaml
loop_count: 10                     # 最多循环10次
```

**说明**:
- 无论退出条件如何,达到最大次数后强制退出
- 默认值: 10
- 建议范围: 5-100
- 设置过大可能导致性能问题

**循环次数统计**:
- 从 0 开始计数
- 第一次执行: 循环次数 = 0
- 第二次执行: 循环次数 = 1
- ...
- 达到 loop_count 时停止

## 退出条件 (break_conditions)

定义何时退出循环的条件列表。

### 基础结构

```yaml
break_conditions:
- varType: "string"                # 变量类型
  variable_selector:               # 变量选择器
  - "节点ID"
  - "路径"
  - "字段名"
  comparison_operator: "="         # 比较运算符
  id: "条件ID"                     # 条件唯一ID
  value: "比较值"                  # 比较值
```

### 变量类型 (varType)

| 类型 | 说明 | 示例 |
|------|------|------|
| `string` | 字符串 | "success", "completed" |
| `number` | 数字 | 100, 3.14 |
| `boolean` | 布尔值 | true, false |
| `object` | 对象 | {"key": "value"} |
| `array` | 数组 | [1, 2, 3] |

### 比较运算符 (comparison_operator)

| 运算符 | 说明 | 适用类型 |
|--------|------|----------|
| `empty` | 为空 | 所有类型 |
| `not empty` | 不为空 | 所有类型 |
| `=` / `is` | 等于 | 字符串、数字、布尔值 |
| `≠` / `is not` | 不等于 | 字符串、数字、布尔值 |
| `>` | 大于 | 数字 |
| `<` | 小于 | 数字 |
| `≥` | 大于等于 | 数字 |
| `≤` | 小于等于 | 数字 |
| `contains` | 包含 | 字符串、数组 |
| `not contains` | 不包含 | 字符串、数组 |

## 逻辑运算符 (logical_operator)

当有多个退出条件时,决定如何组合这些条件。

### and - 与运算

所有条件都满足时才退出循环。

```yaml
logical_operator: "and"
break_conditions:
- comparison_operator: "="
  value: "success"
- comparison_operator: ">"
  value: 100
```

**说明**: 条件1 AND 条件2 都为true时才退出

### or - 或运算

任一条件满足时就退出循环。

```yaml
logical_operator: "or"
break_conditions:
- comparison_operator: "="
  value: "success"
- comparison_operator: "="
  value: "failed"
```

**说明**: 条件1 OR 条件2 任一为true时就退出

## 循环变量 (loop_variables)

用于在循环过程中记录状态或累计值(可选)。

```yaml
loop_variables:
- name: "counter"                  # 变量名
  initial_value: 0                 # 初始值
- name: "sum"
  initial_value: 0
```

**说明**:
- 在循环开始前初始化
- 在循环内可以读取和修改
- 用于跨循环次数传递数据

## 循环开始节点 (loop-start)

循环开始节点是循环内部的起始节点,由系统自动创建。

```yaml
data:
  title: ""                        # 节点标题
  type: "loop-start"               # 节点类型
  selected: false
  desc: ""
  isInLoop: true                   # 是否在循环中
selectable: false                  # 不可选择
sourcePosition: "right"            # 源位置
type: "custom-loop-start"          # 自定义类型
parentId: "1768549904519"          # 父节点ID(循环节点)
draggable: false                   # 不可拖动
targetPosition: "left"             # 目标位置
width: 44
height: 48
id: "1768549904519start"           # ID格式: {循环节点ID}start
```

**注意**: 此节点自动生成,不需要手动配置。

## 循环内节点标记

循环内的所有节点必须设置以下字段:

```yaml
data:
  isInLoop: true                   # 标记节点在循环中
  loop_id: "循环节点ID"            # 循环节点的ID
parentId: "循环节点ID"             # 父节点ID
```

**示例**:
```yaml
# 循环节点
- data:
    type: "loop"
    id: "1768549904519"

# 循环内的节点
- data:
    type: "assigner"
    title: "变量赋值"
    isInLoop: true                 # 必须设置
    loop_id: "1768549904519"       # 指向循环节点
  parentId: "1768549904519"        # 父节点ID
```

## 配置示例

### 场景一:重试机制(最多重试5次)

```yaml
data:
  type: "loop"
  title: "重试调用接口"
  loop_count: 5                    # 最多重试5次
  logical_operator: "or"
  break_conditions:
  # 条件1: 调用成功
  - varType: "boolean"
    variable_selector:
    - "api_call_node_id"
    - "success"
    comparison_operator: "="
    id: "break_cond_1"
    value: "true"
  # 条件2: 已达到最大重试次数(由loop_count控制)
  start_node_id: "loop_node_id_start"
```

**流程**:
```
循环开始
    ↓
调用API接口
    ↓
检查结果
    ↓
├─ 成功: 退出循环(break_conditions满足)
└─ 失败: 继续下一次循环
```

## 退出循环的方式

### 方式1: 满足退出条件

```yaml
# 退出条件配置
break_conditions:
- varType: "boolean"
  variable_selector: ["node_id", "isDone"]
  comparison_operator: "="
  value: "true"
```

**执行流程**:
```
循环开始
    ↓
执行循环内节点
    ↓
检查退出条件
    ↓
├─ 满足条件: 退出循环
└─ 不满足: 继续下一次循环
```

### 方式2: 达到最大循环次数

```yaml
loop_count: 10                     # 最多循环10次
```

**执行流程**:
```
循环次数 = 0
    ↓
执行循环内节点
    ↓
循环次数 + 1
    ↓
├─ 达到loop_count: 强制退出
└─ 未达到: 继续循环
```

### 方式3: 无退出条件(仅依赖loop_count)

```yaml
data:
  type: "loop"
  title: "固定次数循环"
  loop_count: 5
  break_conditions: []             # 空数组,无退出条件
```

**说明**: 总是循环 loop_count 次后退出

## 实际应用场景

### 场景一: API重试机制

```
循环: 最多重试5次
    ↓
    调用外部API
    ↓
    条件判断: 是否成功
    ├─ 成功: 满足退出条件,退出循环
    │   ↓
    │   返回成功结果
    └─ 失败: 不满足退出条件,继续循环
        ↓
        等待延迟(指数退避)
        ↓
        重新调用API
```

**配置要点**:
- `loop_count`: 最大重试次数
- `break_conditions`: API调用成功
- 增加延迟避免频繁调用

```
提交异步任务

    ↓

    等待5秒
    ↓
    查询任务状态
    ↓
    条件判断: 状态
    ├─ completed: 退出循环

    │   获取任务结果
    ├─ failed: 退出循环
    │   ↓

        继续等待
```

## 最佳实践

### 1. 设置合理的loop_count

```yaml
# ✅ 好的做法: 根据场景设置合理值
# API重试
loop_count: 5

# 轮询任务(每5秒一次,最多5分钟)
loop_count: 60

# 分页处理(每批100条,最多处理10000条)
loop_count: 100

# ❌ 避免的做法
loop_count: 1000                   # 过大,可能导致性能问题
loop_count: 1                      # 过小,失去循环意义
```

### 2. 添加延迟避免过载

在循环内添加延迟节点:

```yaml
# 在循环开始处添加延迟
- data:
    type: "code"
    title: "等待延迟"
    isInLoop: true
    loop_id: "loop_node_id"
```

**代码示例**:
```groovy
def main() {
    // 固定延迟
    Thread.sleep(3000)  // 等待3秒
    
    return [delayed: true]
}
```

**指数退避策略**:
```groovy
def main(loopIndex) {
    // 指数退避: 1秒, 2秒, 4秒, 8秒...
    def delay = Math.pow(2, loopIndex) * 1000
    def maxDelay = 30000  // 最大30秒
    delay = Math.min(delay, maxDelay)
    
    println "等待 ${delay}ms"
    Thread.sleep(delay as long)
    
    return [delay: delay]
}
```

### 3. 明确退出条件

```yaml
# ✅ 好的做法: 明确的退出条件
break_conditions:
- varType: "string"
  variable_selector: ["node_id", "status"]
  comparison_operator: "="
  value: "completed"

# ❌ 避免的做法: 依赖隐式条件
break_conditions: []               # 仅依赖loop_count
```

### 4. 记录循环状态

```groovy
def main(loopIndex) {
    println "=== 循环状态 ==="
    println "当前循环次数: ${loopIndex + 1}"
    println "时间: ${new Date()}"
    
    return [
        loopIndex: loopIndex,
        timestamp: new Date().time
    ]
}
```

### 5. 处理循环超时

在循环后检查是否因达到最大次数而退出:

```
循环节点
    ↓
条件判断: 是否达到最大次数
├─ 是: 超时处理
│   ↓
│   记录超时日志
│   ↓
│   返回超时错误
└─ 否: 正常退出
    ↓
    返回成功结果
```

### 6. 使用循环变量记录状态

```yaml
loop_variables:
- name: "retryCount"
  initial_value: 0
- name: "lastError"
  initial_value: ""
- name: "startTime"
  initial_value: 0
```

**在循环内更新**:
```yaml
- data:
    type: "assigner"
    title: "更新状态"
    isInLoop: true
    loop_id: "loop_node_id"
    items:
    - variable_selector: ["loop_node_id", "retryCount"]
      value: "{{#loop_node_id.retryCount#}} + 1"
    - variable_selector: ["loop_node_id", "lastError"]
      value: "{{#error_node_id.message#}}"
```

### 7. 避免死循环

**检查项**:
- ✅ 设置了合理的 `loop_count`
- ✅ 至少有一个 `break_conditions`
- ✅ 退出条件是可达到的
- ✅ 变量在循环内有更新

**示例**:
```yaml
# ✅ 好的做法
loop_count: 10
break_conditions:
- varType: "boolean"
  variable_selector: ["node_id", "isDone"]
  comparison_operator: "="
  value: "true"

# ❌ 危险的做法
loop_count: 999999
break_conditions: []
# 如果内部逻辑有问题,会循环很多次
```

### 8. 循环内避免嵌套循环

```yaml
# ❌ 避免: 循环嵌套循环
循环节点1
└─ 循环节点2
   └─ 业务逻辑

# ✅ 推荐: 使用迭代嵌套
循环节点
└─ 迭代节点
   └─ 业务逻辑
```

**原因**: 循环嵌套循环可能导致执行次数指数增长

### 9. 条件组合策略

**OR组合** (任一条件满足即退出):
```yaml
logical_operator: "or"
break_conditions:
- 条件1: 成功
- 条件2: 失败
- 条件3: 超时
```

**AND组合** (所有条件都满足才退出):
```yaml
logical_operator: "and"
break_conditions:
- 条件1: 数据处理完成
- 条件2: 结果已保存
- 条件3: 通知已发送
```

## 调试技巧

### 1. 输出循环信息

```groovy
def main(loopContext) {
    def loopIndex = loopContext?.loopIndex ?: 0
    def maxCount = loopContext?.maxCount ?: 0
    
    println "=== 循环调试信息 ==="
    println "当前循环: ${loopIndex + 1}/${maxCount}"
    println "时间: ${new Date()}"
    println "进度: ${(loopIndex + 1) / maxCount * 100}%"
    
    return [
        loopIndex: loopIndex,
        maxCount: maxCount,
        progress: (loopIndex + 1) / maxCount * 100
    ]
}
```

### 2. 检查退出条件

在循环内添加代码节点检查条件值:

```groovy
def main(checkValue) {
    println "=== 退出条件检查 ==="
    println "条件值: $checkValue"
    println "条件类型: ${checkValue?.getClass()?.name}"
    println "是否满足退出: ${checkValue == 'completed'}"
    
    return [
        conditionValue: checkValue,
        shouldBreak: checkValue == 'completed'
    ]
}
```

### 3. 限制循环次数(测试)

在测试阶段减少循环次数:

```yaml
# 生产环境
loop_count: 60

# 测试环境
loop_count: 3                      # 只循环3次测试逻辑
```

### 4. 记录每次循环的结果

```groovy
def main() {
    // 获取或初始化结果列表
    def results = context.get("loopResults") ?: []
    
    // 添加当前循环结果
    results << [
        index: results.size(),
        timestamp: new Date().time,
        status: "processed"
    ]
    
    // 保存回context
    context.set("loopResults", results)
    
    println "已完成 ${results.size()} 次循环"
    
    return [results: results]
}
```

## 常见问题

### 1. 循环一直不退出

**可能原因**:
- 退出条件永远不满足
- 变量路径错误
- 数据类型不匹配
- loop_count设置过大

**排查步骤**:
1. 在循环内输出退出条件的实际值
2. 检查变量路径是否正确
3. 验证数据类型是否匹配
4. 临时降低loop_count测试

**解决示例**:
```groovy
// 在循环内添加
def main(conditionValue) {
    println "退出条件当前值: $conditionValue"
    println "期望值: 'completed'"
    println "是否相等: ${conditionValue == 'completed'}"
    println "实际类型: ${conditionValue?.getClass()?.name}"
    
    return [value: conditionValue]
}
```

### 2. 循环只执行一次就退出

**可能原因**:
- 退出条件在第一次就满足
- 退出条件配置错误
- loop_count设置为1

**排查**:
```groovy
def main() {
    println "循环开始执行"
    println "loop_count: ..."
    println "退出条件: ..."
    return [started: true]
}
```

### 3. 循环内变量无法访问

**问题**: 在循环内节点无法访问循环变量

**原因**: 未正确设置 `isInLoop` 和 `loop_id`

**解决**:
```yaml
# ✅ 正确配置
data:
  type: "assigner"
  isInLoop: true                   # 必须设置
  loop_id: "循环节点ID"            # 必须指向循环节点
parentId: "循环节点ID"             # 必须设置

# ❌ 错误配置
data:
  type: "assigner"
  # 缺少 isInLoop 和 loop_id
```

### 4. 多个退出条件不生效

**问题**: 设置了多个OR条件,但仍不退出

**原因**: logical_operator设置错误

**解决**:
```yaml
# 任一条件满足即退出
logical_operator: "or"             # 使用OR

# 所有条件都满足才退出
logical_operator: "and"            # 使用AND
```

### 5. 循环性能问题

**问题**: 循环执行很慢

**优化策略**:
1. 减少循环内节点数量
2. 添加适当延迟
3. 优化循环内的查询操作
4. 使用批量操作代替循环

**示例**:
```yaml
# ❌ 不好的做法
循环
└─ 查询单条数据
   └─ 处理数据
   └─ 插入数据

# ✅ 好的做法
查询所有数据(一次性)
└─ 迭代
   └─ 处理数据
└─ 批量插入
```

## 循环 vs 迭代

| 特性 | 循环 (loop) | 迭代 (iteration/iteration-vue) |
|------|-------------|--------------------------------|
| 执行方式 | 条件驱动 | 数据驱动 |
| 退出条件 | 动态判断 | 遍历完所有元素 |
| 执行次数 | 不确定(取决于条件) | 确定(数组长度或范围) |
| 适用场景 | 重试、轮询、等待 | 批量处理、数组遍历 |
| 提前退出 | 支持(break_conditions) | 不支持 |
| 循环变量 | loop_variables | 迭代变量 |
| 死循环风险 | 有(需设置loop_count) | 无 |

**选择建议**:

**使用循环的场景**:
- API调用重试
- 异步任务轮询
- 等待资源就绪
- 需要根据结果决定是否继续
- 不确定循环次数

**使用迭代的场景**:
- 遍历数组元素
- 批量数据处理
- 明确的次数范围
- 不需要提前退出
- 数据驱动的处理

**组合使用**:
```
循环: 分页查询
    ↓
    查询当前页数据
    ↓
    迭代: 处理当前页的每条数据
    │   ↓
    │   处理单条记录
    ↓
    检查是否还有下一页
    ↓
    └─ 有: 继续循环
       无: 退出循环
```

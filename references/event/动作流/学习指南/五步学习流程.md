# AI 生成 DSL 规范指南 — 五步学习流程

**本文档为 AI 助手提供严格的 DSL 生成规范，必须完全遵守。**

> 本文档从 `event/动作流/index.md` 拆分而来，便于按需加载。

---

## 📖 学习顺序（严格按此顺序）

### 第一步：掌握基础结构（必须完全遵循）

**⚠️ 警告**：基础结构是 DSL 的根基，任何层次错误、字段缺失或格式不规范都会导致 DSL 无法解析。

**必须学习的文档（使用 `read_document` 查询）**：
1. `read_document(path="动作流/基础结构/app配置.md")` - 应用级配置，**字段和层次结构必须完全一致**
2. `read_document(path="动作流/基础结构/workflow配置.md")` - 工作流配置，**特别注意 features、graph 的嵌套层次**
3. `read_document(path="动作流/基础结构/graph结构.md")` - 节点和连接结构，**nodes 和 edges 的格式必须严格遵循**

**生成要求**：
- ✅ **必须**：YAML 文件以 `---` 开头
- ✅ **必须**：**顶层结构顺序：`app` → `workflow` → `kind` → `version` → `dependencies`**（app 必须在 workflow 之前！）
- ✅ **必须**：`kind: "app"` 和 `version: "3.0"` 放在文件末尾，后面跟 `dependencies: []`
- ✅ **必须**：严格遵循字段嵌套层次（如 `app` → `workflow` → `graph` → `nodes`/`edges`）
- ✅ **必须**：所有节点的 `id` 必须唯一，建议使用时间戳
- ✅ **必须**：所有 `edges` 的 `source` 和 `target` 必须指向实际存在的节点 `id`
- ❌ **禁止**：随意调整字段层次或创造不存在的字段
- ❌ **禁止**：遗漏必需字段（如 `type`、`data`、`position` 等）

### 第二步：理解节点能力

**必须学习的节点类型（使用 `read_document` 查询）**：

1. **触发节点** - 流程入口
   - 使用 `read_document(path="动作流/节点/触发节点/webhook触发.md")` - **特别注意 requestBody 的嵌套结构规范**
   - 使用 `read_document(path="动作流/节点/触发节点/定时任务触发.md")`

2. **逻辑节点** - 流程控制
   - 使用 `read_document(path="动作流/节点/逻辑节点/条件分支.md")` - 必须理解 `true`/`false` 分支的连接方式
   - 使用 `read_document(path="动作流/节点/逻辑节点/迭代.md")` - 必须理解双层连接结构

3. **转换节点** - 数据处理
   - 使用 `read_document(path="动作流/节点/转换节点/代码执行.md")` - 使用 Groovy 脚本
   - 使用 `read_document(path="动作流/节点/转换节点/变量赋值.md")`
   
4. **结束节点** - 流程出口
   - 使用 `read_document(path="动作流/节点/结束节点/结束.md")` - 必须正确构造返回数据结构

**生成要求**：
- ✅ **必须**：每个节点的 `data.type` 与节点功能对应
- ✅ **必须**：理解节点的输入输出，正确使用变量引用格式
- ✅ **必须**：条件判断节点的所有分支必须有连线
- ✅ **必须**：迭代节点必须配置双层连线（迭代 → 后续节点，迭代开始 → 首个子节点）
- ❌ **禁止**：使用不存在的节点类型
- ❌ **禁止**：节点缺少必要的连线（除开始节点无入站、结束节点无出站）

### 第三步：掌握资产使用（严格遵循优先级）

**⚠️ 重要变更**：本步骤需要结合 MCP 工具动态查询资产信息，不再依赖静态文档路径。

**查询资产的方法**：

1. **查询业务动作通用说明**
   - 使用 `read_document` 工具查询：`动作流/节点/业务动作/通用节点说明.md`
   - 目的：了解业务动作节点的通用配置规范、参数映射规则、最佳实践等

2. **查询具体业务动作（资产）**
   - 使用 `query_action` 工具搜索具体的业务动作
   - 示例：`query_action(query="部门服务")` 或 `query_action(query="用户")`
   - 目的：获取具体业务动作的工具名称、参数配置、使用示例等
   - **如果查询结果包含 `panelPath` 字段**：说明这是有面板配置的业务动作节点，需要根据 `panelPath` 查询对应的面板配置文档

3. **根据 panelPath 查询面板配置**（当资产包含 panelPath 时）
   - 数据表节点：使用 `read_document(path="动作流/节点/业务动作/基础支撑/数据表/面板配置说明.md")`
   - 根据返回的 `panelPath` 值，找到对应的面板配置（如 `frame-datasource/data-query-list` 对应查询列表）
   - 获取该面板的 `action`、`tool_name`、`formData` 参数列表等配置信息
   - 按照配置模板生成完整的节点配置

**资产使用优先级（从高到低）**：

1. **第一优先级：业务专有资产** 🔴
   - **查询方式**：使用 `query_action` 工具查询相关业务动作（如搜索"部门"、"用户"、"角色"等关键词）
   - **典型资产示例**（通过 `query_action` 查询返回）：
     - 部门服务相关：`methodName: "新增部门"`、`methodName: "新增部门信息"`、`methodName: "查询部门基本信息"`、`methodName: "查询部门集合"` 等
     - 用户服务相关：`methodName: "新增用户"`、`methodName: "新增用户（不包含照片）"`、`methodName: "根据用户字段查询出用户"` 等
     - 角色服务相关：角色管理相关的业务动作（通过 `query_action(query="角色")` 查询）
   - **规则**：如果业务需求与这些服务相关（如创建部门、用户、角色），**必须优先使用业务专有资产**，不得降级使用通用工具
   - **⚠️ 重要**：资产的 `methodName` 为中文名称，查询结果会包含完整的 `inparam`（输入参数）和 `outparam`（输出参数）结构

2. **第二优先级：通用数据表资产** 🟡
   - **查询方式**：使用 `query_action` 工具查询通用数据操作（如搜索"新增数据"、"查询"等关键词）
   - **典型资产示例**（通过 `query_action` 查询返回）：
     - 数据操作类：`methodName: "新增数据"`、`methodName: "查询数量"`、`methodName: "查询list"`、`methodName: "分页查询"` 等
   - **规则**：仅当业务专有资产无法满足需求时，才使用通用数据表资产
   - **⚠️ 重要：参数配置必须参考场景示例或面板配置文档**（详见下方"资产参数配置来源优先级"）

3. **第三优先级：代码节点** 🟢
   - **查询方式**：使用 `read_document` 工具查询：`动作流/节点/转换节点/代码执行.md`
   - **规则**：仅当资产无法满足需求时，才使用代码节点编写 Groovy 脚本

**📋 资产参数配置来源优先级（严格遵循）**

⚠️ **核心原则**：资产的参数配置**必须有明确来源**，禁止自行编造参数名或格式。

| 优先级 | 来源 | 说明 | 示例 |
|-------|------|------|------|
| 🔴 **第一** | **场景示例文档** | 如果场景示例中有该资产的完整配置，**必须严格照搬**，不得修改参数名或添加/删除参数 | `Push模式-接收数据-多层级插入.md` 中的数据表插入配置 |
| 🟡 **第二** | **面板配置文档** | 如果有对应的面板配置文档，按文档中的参数列表配置 | `动作流/节点/业务动作/基础支撑/数据表/面板配置说明.md` |
| 🟢 **第三** | **MCP query_action 查询结果** | 仅当上述文档都没有明确说明时，才使用 `query_action` 返回的 `inparam` 作为参数配置依据 | `query_action(query="新增数据")` 返回的参数结构 |

**⚠️ 特别说明：通用数据表资产参数配置**

通用数据表资产（新增数据、查询详情、查询列表等）的参数配置，**必须参考场景示例**，而非 `query_action` 返回结果：

| 操作类型 | 正确参数名 | ❌ 常见错误 | 来源文档 |
|---------|-----------|------------|----------|
| 新增数据 | `sqlTableName` | ❌ `tableName` | 场景示例 |
| 查询详情 | `sqlTableName` | ❌ `tableName` | 场景示例 |
| 查询列表 | `sqlTableName` | ❌ `tableName` | 场景示例 |
| 新增数据 | 无需 `appGuid` | ❌ 添加 `appGuid: ""` | 场景示例 |
| 查询详情 | 无需 `sortFields` | ❌ 添加 `sortFields: []` | 场景示例 |

**正确配置示例（来自场景示例）**：
```yaml
# 新增数据节点
formData:
- name: "sqlTableName"      # ✅ 正确：sqlTableName
  valueMode: "fixed"
  value: "CG_ProjectInfo"
- name: "entities"          # ✅ 直接是entities，无appGuid
  valueMode: "fixed"
  value: [...]

# 查询详情节点
formData:
- name: "sqlTableName"      # ✅ 正确：sqlTableName
  valueMode: "fixed"
  value: "CG_ProjectInfo"
- name: "conditions"        # ✅ 直接是conditions，无appGuid
  valueMode: "fixed"
  value: {...}
- name: "queryFields"       # ✅ 无sortFields
  valueMode: "fixed"
  value: [...]
```

**❌ 错误配置示例（来自 query_action 返回但与实际不符）**：
```yaml
# 错误！不要照搬 query_action 返回的参数名
formData:
- name: "tableName"         # ❌ 错误：应为 sqlTableName
  valueMode: "fixed"
  value: "CG_ProjectInfo"
- name: "appGuid"           # ❌ 错误：场景示例中无此参数
  valueMode: "fixed"
  value: ""
```

**生成要求**：
- ✅ **必须**：首先使用 `read_document(path="动作流/节点/业务动作/通用节点说明.md")` 了解业务动作节点的通用配置规范
- ✅ **必须**：使用 `query_action` 工具查询业务需求相关的资产，优先选择业务专有资产
- ✅ **必须**：查询到业务专有资产后（如 `methodName: "新增部门"`、`methodName: "新增用户"`），必须优先使用
- ✅ **必须**：仅在业务专有资产不存在时，才降级查询和使用通用资产（如 `methodName: "新增数据"`）
- ✅ **必须**：仅在资产无法满足时，才使用代码节点
- ✅ **必须**：使用资产前，通过 `query_action` 获取详细信息，从返回结果的 `inparam` 和 `outparam` 了解参数格式和返回值结构
- ✅ **必须**：所有资产信息必须来自 `query_action` 查询结果，不得依赖静态文档路径或假设资产名称
- ✅ **必须**：资产的 `methodName` 为中文名称，在配置节点时使用查询返回的准确名称
- ❌ **禁止**：未查询就假设资产不存在
- ❌ **禁止**：查询到业务专有资产却使用通用资产
- ❌ **禁止**：查询到资产却使用代码节点实现相同功能
- ❌ **禁止**：编造不存在的资产名称或使用错误的 methodName 格式

### 第四步：查找匹配场景（业务逻辑参考）

**⚠️ 重要**：场景示例用于理解业务逻辑和流程设计，但**不能替代节点文档**。

**查找流程**：
1. **分析需求类型**：
   - Push 模式：外部向系统推送数据
   - Pull 模式：系统主动拉取外部数据
   - Schedule 模式：定时任务自动执行

2. **查找场景示例**：
   - 使用 `read_document` 工具查询场景示例文档：
     - `动作流/场景示例/Push模式-接收数据-多层级插入.md`
     - `动作流/场景示例/Pull模式-输出数据-多表联查.md`
     - `动作流/场景示例/Schedule模式-定时同步-组织架构.md`
   - 查看是否有类似的业务场景

3. **决策**：
  - ✅ **找到匹配场景**：学习其业务实现思路，但**仍需查询节点文档确保配置完整**
   - ❌ **未找到匹配场景**：基于基础文档从零构建

**学习重点与使用规范**：

1. **业务实现思路**（从场景示例学习）
   - 业务流程顺序（如：验证 → 查询 → 判断 → 插入/更新 → 返回）
   - 错误处理机制（如：入参不合法如何返回错误）
   - 数据校验逻辑（如：先判断部门是否存在，再处理用户）
   - 事务边界考虑（如：部门和用户是否需要同时成功）
   - 节点连接顺序和流程控制

2. **节点配置结构**（**必须**查询节点文档）
   - ⚠️ **核心原则**：场景示例可能缺少节点必需字段，**必须**查询节点文档补全
   - 节点必需字段（如 `id`、`type`、`data`、`position`）
   - 节点特殊属性（如迭代节点的 `resultType`、`zIndex`、`start_node_id`）
   - 连线配置（如 `sourceHandle`、`data.isInIteration`、`data.iteration_id`）
   - **查询方法**：使用 `read_document` 查询节点文档（如 `动作流/节点/逻辑节点/迭代.md`）

3. **资产参数配置**（优先级：场景示例 > 面板配置 > query_action）
   - 资产使用模式和参数配置（通过 `query_action` 查询具体资产信息）
   - 变量引用方式和数据传递
   - Webhook requestBody 结构（Push 模式）

**⚠️ 常见错误**：
- ❌ **错误做法**：只参考场景示例，不查询节点文档 → 导致节点字段缺失、导入报错
- ✅ **正确做法**：场景示例学习业务逻辑，节点文档确保配置完整 → 生成完整可用的 DSL

### 第五步：参考数据字典（辅助资料）

**使用场景**：
- 当需要使用通用数据表资产（`methodName: "新增数据"`）操作数据库时
- 需要了解表结构、字段类型、字段含义、主键、必填字段等信息时
- 需要进行数据映射和参数配置时

**查询数据字典的方法**：

1. **使用 `query_table` 工具**（实时、准确）：
   - 查询精确表名：`query_table(query="Frame_Ou")` - 返回 `FRAME_OU` 表完整结构
   - 查询精确表名：`query_table(query="Frame_User")` - 返回 `FRAME_USER` 和 `FRAME_USER_EXTENDINFO` 表结构
   - 查询关键词：`query_table(query="部门")` - 返回所有包含"部门"的表
   - 查询关键词：`query_table(query="用户")` - 返回所有包含"用户"的表

**查询结果包含的关键信息**：
- `tableName` - 表英文名（用于通用资产的 `tableName` 参数）
- `tableCName` - 表中文名（业务含义）
- `columns[].name` - 字段英文名（数据映射字段名）
- `columns[].cname` - 字段中文名（字段含义）
- `columns[].type` - 字段数据类型（nvarchar、Integer、DateTime 等）
- `columns[].length` - 字段长度
- `columns[].isPrimaryKey` - 是否主键（主键字段需生成 UUID）
- `columns[].required` - 是否必填（必填字段不能为空）

**使用规则**：
- ✅ **必须**：使用通用数据表资产前，通过 `query_table` 查询字段结构
- ✅ **必须**：使用查询结果中的实际字段名（`columns[].name`），避免拼写错误
- ✅ **必须**：注意主键字段（`isPrimaryKey: true`）和必填字段（`required: true`）
- ✅ **建议**：通过 `columns[].cname` 和 `tableCName` 理解业务含义
- ⚠️ **注意**：数据字典仅在使用通用资产时参考，优先使用业务专有资产
